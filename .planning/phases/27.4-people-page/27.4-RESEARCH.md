# Phase 27.4: People Page - Research

**Researched:** 2026-03-18
**Domain:** React + Flask — People directory UI with split layout and per-person detail
**Confidence:** HIGH

## Summary

Phase 27.4 adds a third tab ("People") to the top tab bar. The page has a split layout: a filterable table of person notes on the left, a read-only per-person detail view on the right. The detail view renders the person's note body (markdown) plus four collapsible sections: Meetings, Backlinks, Open Action Items.

All necessary frontend infrastructure (TabBar, UIContext, shadcn Table, react-markdown, ActionsPage data patterns) and backend data (backlinks via `/api/notes/<path>/meta`, actions via `/api/actions?assignee=`, notes list via `/api/notes`) already exist. The backend needs one new endpoint: `/api/people` to return people notes with aggregated stats (open action count, last note date) — this avoids N+1 fetches from the frontend.

The implementation follows the same patterns as `ActionsPage.tsx` (direct `fetch()`, `useState`/`useEffect`, shadcn Table) and `NoteViewer.tsx` (react-markdown). No new dependencies are required.

**Primary recommendation:** One new backend endpoint (`GET /people`) + one new frontend component (`PeoplePage.tsx`) following ActionsPage/NoteViewer patterns. Wire into UIContext, TabBar, and App.tsx.

### Pattern 2: New Backend Endpoint `GET /people`

Required because the frontend needs aggregated stats (open action count, last note date) per person. Computing this client-side would require N+1 fetches.

```python
# engine/api.py
@app.get("/people")
def list_people():
    conn = get_connection()
    conn.row_factory = sqlite3.Row
    brain_root = Path(os.environ.get("BRAIN_PATH", os.path.expanduser("~/SecondBrain"))).resolve()
    people_prefix = str(brain_root / "people") + "/"
    rows = conn.execute(
        "SELECT n.path, n.title, n.updated_at, "
        "  (SELECT COUNT(*) FROM action_items a WHERE a.assignee_path=n.path AND a.done=0) AS open_actions "
        "FROM notes n WHERE n.path LIKE ? ORDER BY n.title",
        (people_prefix + "%",)
    ).fetchall()
    conn.close()
    return jsonify({"people": [dict(r) for r in rows]})
```

This uses the `people/` folder prefix as canonical filter — consistent with locked decision. `updated_at` serves as "last note date".

### Pattern 4: Collapsible Sections

The existing shadcn install may not include `Collapsible`. Use native HTML `<details>/<summary>` (zero-dep, WKWebView safe, same approach as tag autocomplete datalist in Phase 23) or use a simple `useState` boolean per section.

Recommended: simple `useState` boolean with chevron icon — keeps full control over styling and item count display in header. Avoids depending on whether `Collapsible` was generated.

```tsx
function Section({ title, count, children }: { title: string; count: number; children: React.ReactNode }) {
  const [open, setOpen] = useState(true)
  return (
    <div className="border-b">
      <button
        className="flex w-full items-center justify-between px-4 py-2 text-sm font-medium"
        onClick={() => setOpen(o => !o)}
      >
        <span>{title} ({count})</span>
        <ChevronDown className={cn("h-4 w-4 transition-transform", open && "rotate-180")} />
      </button>
      {open && <div className="px-4 pb-3">{children}</div>}
    </div>
  )
}
```

### Pattern 5: "Open in Notes" Navigation

Use `useUIContext().setCurrentView('notes')` + `useNoteContext().setSelectedPath(path)` or equivalent. Look at how ActionsPage or NoteViewer navigate — but since PeoplePage is read-only and note editing is in the Notes tab, a button that calls `setCurrentView('notes')` and then triggers note selection is the right pattern. The exact mechanism depends on NoteContext's API surface.

Check `NoteContext.tsx` during implementation. From NoteViewer we can see `setIsDirty` exists; likely `currentPath` is set externally from sidebar click. A simpler approach: navigate to notes tab, relying on the user to find the note. Or: expose a `openNote(path)` action from NoteContext that the People page can call.

### Pattern 6: Meetings Derivation

Meetings for a person come from `/api/notes/<path>/meta` → `people` field OR from backlinks in meetings/ folder. The existing `/api/notes/<path>/meta` endpoint returns `backlinks` (all notes referencing this person by title) and `people` (people mentioned in the selected note). It does NOT directly return meetings that list this person in their `people` column.

Two implementation options:
1. Filter backlinks (from `/meta`) by path containing `meetings/` → LOW N queries, uses existing endpoint
2. New query in `/api/people/<path>` that joins on the `notes.people` JSON column → accurate

**Recommended:** Option 1 for simplicity. In the detail fetch, call `/api/notes/<path>/meta` and split backlinks into meetings (path contains `/meetings/`) and other backlinks. This matches the CONTEXT.md description "via `notes.people` column or tag match" and requires zero new backend code for this sub-feature.

### Anti-Patterns to Avoid
- **N+1 fetches for people list stats:** Don't compute open_actions count per person in frontend with N action fetches. Use the `/api/people` endpoint with aggregated SQL.
- **Reusing NoteViewer for person detail:** NoteViewer is for editing-capable note display. Person detail is read-only with a different section structure — build `PersonDetail` as a separate component.
- **Using `type === 'people'` as filter in `/api/people`:** The `type` field in frontmatter may not always be set to "people" even for files in `people/`. Use the `people/` folder prefix on `path` as the canonical filter — per locked decision.

## Common Pitfalls

### Pitfall 1: `type View` union not updated before TabBar or App.tsx changes
**What goes wrong:** TypeScript errors cascade — TabBar `setCurrentView(tab.id)` won't typecheck with `'people'` if the union hasn't been updated.
**Why it happens:** UIContext.tsx is the single source of truth for View type; all callers depend on it.
**How to avoid:** Update UIContext.tsx first (Wave 1 task order). The compile error will immediately surface if missed.

### Pitfall 2: `people/` path prefix varies between test brain and real brain
**What goes wrong:** Test uses `tmp_path/brain/people/` but the LIKE query uses the BRAIN_PATH env var. If BRAIN_PATH is not patched in tests, the prefix will be wrong and return 0 rows.
**Why it happens:** `_resolve_note_path` and `_get_prefs_path` both use `os.environ.get("BRAIN_PATH", ...)` at call time. Tests must set `BRAIN_PATH` env var AND patch `engine.db.DB_PATH` / `engine.paths.DB_PATH`.
**How to avoid:** Follow the Phase 24 pattern: `gui_brain` fixture sets `os.environ["BRAIN_PATH"]` before Flask starts. For unit tests of `/api/people`: use `client` fixture that patches both DB_PATH refs.

### Pitfall 3: `notes.people` JSON column contains display names not paths
**What goes wrong:** Trying to match meetings by `notes.people` column using the person's file path (e.g. `/brain/people/alice.md`) fails — the column stores name strings like `["Alice"]`, not paths.
**Why it happens:** The `people` column is populated by entity extraction (`extract_entities()`), which extracts name strings.
**How to avoid:** For meetings, filter the backlinks returned by `/meta` by path containing `/meetings/` instead of querying the `people` column directly. This is the simplest correct approach.

### Pitfall 4: `list_actions` `done` parameter semantics
**What goes wrong:** `list_actions(conn, done=False)` returns OPEN items (done=0). The frontend wants open items for the person detail. Passing `done=True` would return completed items — wrong for this use case.
**Why it happens:** The `done` param means "return items where done matches this value", not "include done items".
**How to avoid:** Call `GET /actions?assignee=<path>` with no `done` param (defaults to `done=0` in the route handler) to get open items for the person.

### Pitfall 5: Encoding person path in fetch URLs
**What goes wrong:** Person paths contain `/` and spaces; fetching `/api/notes/people/alice smith.md/meta` without encoding produces a 404.
**Why it happens:** Flask path converter requires URL-encoded paths; the `_SlashNormMiddleware` handles double-slash but not unencoded spaces.
**How to avoid:** Use `encodeURIComponent(path)` when building fetch URLs — same pattern as NoteViewer: `fetch(\`${getAPI()}/notes/${encodeURIComponent(note.path)}/attachments\`)`.

### Person detail data fetch
```tsx
// Three parallel fetches when a person is selected
useEffect(() => {
  if (!selectedPath) return
  const enc = encodeURIComponent(selectedPath)
  Promise.all([
    fetch(`${getAPI()}/notes/${enc}`).then(r => r.json()),
    fetch(`${getAPI()}/notes/${enc}/meta`).then(r => r.json()),
    fetch(`${getAPI()}/actions?assignee=${enc}`).then(r => r.json()),
  ]).then(([note, meta, acts]) => {
    setPersonNote({ body: note.body ?? '', title: note.title ?? '' })
    const bl = meta.backlinks ?? []
    setMeetings(bl.filter((b: {path: string}) => b.path.includes('/meetings/')))
    setBacklinks(bl.filter((b: {path: string}) => !b.path.includes('/meetings/')))
    setActions(acts.actions ?? [])
  }).catch(() => {})
}, [selectedPath])
```

## Open Questions

1. **"Open in Notes" navigation mechanism**
   - What we know: `useUIContext().setCurrentView('notes')` switches tabs. NoteContext exposes `currentPath`/`currentNote` but it's unclear if there's a `selectNote(path)` action without reading NoteContext.tsx.
   - What's unclear: How does a non-sidebar component programmatically select a note in the Notes tab?
   - Recommendation: Read `NoteContext.tsx` at implementation time (plan 27.4-03). If no `selectNote` action exists, add one (simple `setCurrentPath` exposed from context). This is a small addition, not a new dependency.

2. **`updated_at` as "Last Note date" display format**
   - What we know: The `notes` table has `updated_at` as ISO string.
   - What's unclear: Should display be `2026-03-15` (date only) or relative ("3 days ago")?
   - Recommendation: Date-only format (`updated_at.slice(0, 10)`) — no date library needed, consistent with due_date display in ActionsPage.
