# Phase 27.4: People Page - Context

**Gathered:** 2026-03-18
**Status:** Ready for planning

<domain>
## Phase Boundary

A dedicated People tab in the top tab bar showing: (1) a people directory table and (2) a per-person detail view with the person's note body, linked meetings, backlinks, and open action items. Creating/editing person notes and search across people are out of scope — this phase is a read-only reference surface.

</domain>

<decisions>
## Implementation Decisions

### People directory layout
- Table layout (not cards or list) with columns: Name, Open Action Items count, Last Note date
- Text filter input above the table to filter by name as you type
- Source: notes in `people/` folder are canonical "persons" — `people/` folder and `type: person` are the same thing in practice (both set at capture time); use `people/` folder as the authoritative filter

### Person detail view placement
- Split layout: people table on the left, person detail fills the main content area on the right
- No person selected → empty main area with "Select a person to view details" prompt
- Clicking a person shows a purpose-built summary (not the raw note editor)
- Read-only — includes a button/link to open the note in the Notes tab for editing

### Person detail sections
- Four collapsible sections, each showing item count in the header (e.g. "Meetings (3)")
- Order: **Note body → Meetings → Backlinks → Actions**
  1. **Note body** — rendered markdown of the person note
  2. **Meetings** — notes from `meetings/` folder that mention this person (via `notes.people` column or tag match)
  3. **Backlinks** — all other notes that reference this person (via existing backlinks API)
  4. **Open action items** — action items assigned to this person via `assignee_path`; displayed as list with checkbox, task text, and due date (same style as Action Items page rows)

### Claude's Discretion
- Exact column widths and table density
- Loading/skeleton states
- Empty section messages (e.g. "No meetings yet")
- How the "open in Notes" link/button is styled
- Whether action items in the detail view are interactive (can mark done) or display-only

</decisions>

### Reusable Assets
- `TabBar.tsx`: Existing TABS array with `notes` and `actions` — add `people` tab following same pattern; `UIContext.currentView` type needs `'people'` added
- `ActionsPage.tsx`: Already fetches people list from API for assignee filter (`/api/notes?type=person` or similar) — reuse this for the people directory
- `Table`, `TableHeader`, `TableBody`, `TableRow`, `TableHead`, `TableCell` from `components/ui/table.tsx` — use for people directory table
- `RightPanel.tsx`: Existing right-panel pattern for reference — person detail uses main content area instead but same visual approach
- `NoteViewer.tsx`: Markdown rendering with `react-markdown` + `remark-gfm` — reuse for person note body section
- `Badge` from `components/ui/badge.tsx` — use for action item counts

### Established Patterns
- `useUIContext()` / `UIContext`: `currentView` drives which page renders in App.tsx — add `'people'` view
- `useNoteContext()`: Notes loaded globally; person notes are a subset, filter by folder
- Data fetching: direct `fetch()` calls to Flask API (port 5001), no React Query — follow same pattern as `ActionsPage`
- shadcn/ui + Tailwind: all components use this stack

### Integration Points
- `App.tsx`: Add `currentView === 'people' && <PeoplePage />` render branch alongside existing `<ActionsPage />`
- `TabBar.tsx`: Add `{ id: 'people', label: 'People' }` to TABS array
- `UIContext.tsx`: Add `'people'` to the view type union
- Backend: `/api/notes` with folder filter for people list; `/api/notes/<path>/backlinks` for backlinks; `/api/action_items` filtered by `assignee_path` for person's actions
- `engine/api.py`: May need new endpoint or query param to list people with aggregated stats (open action count, last note date)
