# Phase 4.1: Native macOS UX — Research

**Researched:** 2026-03-14
**Domain:** macOS system integration — `uv tool install`, launchd plist, git hook installer
**Confidence:** HIGH

## Summary

Phase 4.1 delivers three friction-removing integrations that make the second-brain system feel native
on macOS rather than a project-directory-bound tool. All three subproblems are well-understood macOS
patterns with reliable tooling: `uv tool install --editable` for global CLI, `~/Library/LaunchAgents`
plists for per-user daemons, and a Python installer script for one-command git hook adoption.

The key insight across all three: the project already uses `uv` as its build tool and has all
`[project.scripts]` entry points defined. Global CLI is therefore a one-command operation
(`uv tool install --editable .` from the repo root). The launchd plist must specify the full
absolute path to the `uv` binary and set `PATH`/`WorkingDirectory` via `EnvironmentVariables`
because launchd launches with a minimal environment. The git hook installer is a small Python script
that runs `git config core.hooksPath` in each target repo — identical to the approach already used
inside the second-brain repo itself.

**Primary recommendation:** Build a single `scripts/install_native.py` that performs all three
installations in one idempotent run. Each subcommand (global CLI, launchd, git-hooks) is also
callable independently.

## Standard Stack

### Core
| Library/Tool | Version | Purpose | Why Standard |
|---|---|---|---|
| `uv tool install` | uv >=0.4 (current: on PATH) | Install project scripts globally | Native uv feature; replaces pipx; already in use |
| `launchd` + `.plist` | macOS built-in | Run `sb-watch` at login, restart on crash | Only supported mechanism for user-level daemons on macOS |
| `launchctl bootstrap` | macOS built-in | Load/unload plist without reboot | Modern API (replaces deprecated `launchctl load`) |
| `git config core.hooksPath` | git >=2.9 | Point any repo at shared `.githooks/` | Standard approach, already in use within the brain repo |

### Supporting
| Library/Tool | Version | Purpose | When to Use |
|---|---|---|---|
| `subprocess` (stdlib) | 3.12 | Run `uv tool install`, `git config`, `launchctl` | Installer script calling external commands |
| `pathlib` (stdlib) | 3.12 | Path construction for plist, bin dir | Already project-wide convention (FOUND-12) |
| `plistlib` (stdlib) | 3.12 | Write `~/Library/LaunchAgents/*.plist` | Avoids XML string templating errors |

### Alternatives Considered
| Instead of | Could Use | Tradeoff |
|---|---|---|
| `uv tool install --editable` | symlink to `~/.local/bin/sb` | Editable install is cleaner, handles all entry points; symlink requires manual sync when pyproject.toml changes |
| `plistlib` for plist generation | raw XML string template | `plistlib` is stdlib, type-safe, no escaping bugs |
| `launchctl bootstrap` | `launchctl load` (deprecated) | `bootstrap` is the modern macOS API since macOS 10.10; `load` still works but prints deprecation warnings |

### Pattern 1: Global CLI via uv tool install --editable

**What:** `uv tool install --editable .` builds an isolated venv under
`~/.local/share/uv/tools/second-brain/`, creates shims in `~/.local/bin/` for every
`[project.scripts]` entry point, and because it is editable, code changes in `engine/` are
immediately reflected without reinstalling.

**When to use:** Once, during initial setup or after `pyproject.toml` entry points change.

**Key facts (verified from `uv tool install --help` on this machine):**
- Tool bin dir: `~/.local/share/uv/tools/` (confirmed via `uv tool dir`)
- Shims land in `~/.local/bin/` (confirmed: `claude`, `uv`, `uvx` already there)
- `--editable` flag: supported, reflected immediately in source changes
- `--force` flag: required for reinstall if already installed
- `uv tool update-shell`: patches shell RC files if `~/.local/bin` not yet in PATH

```python
# Source: uv tool install --help (verified on this machine)
import subprocess, sys
result = subprocess.run(
    ["uv", "tool", "install", "--editable", "--force", str(repo_root)],
    cwd=repo_root,
    capture_output=True, text=True,
)
if result.returncode != 0:
    print(result.stderr, file=sys.stderr)
    sys.exit(1)
print("Global CLI installed. Entry points: sb-init, sb-reindex, sb-capture, "
      "sb-search, sb-watch, sb-check-links")
```

**Pitfall:** Entry point names defined in `[project.scripts]` must already exist in
`pyproject.toml` before calling `uv tool install`. Existing ones: `sb-init`, `sb-reindex`,
`sb-capture`, `sb-search`, `sb-watch`, `sb-check-links`. No new entry point needed for this phase —
a new `sb` convenience alias or `sb-install` entry point can be added to `pyproject.toml` if desired.

### Pattern 2: launchd User Agent for sb-watch

**What:** A `.plist` file in `~/Library/LaunchAgents/` tells launchd to run `sb-watch` at login
and restart it if it exits. This replaces "run `sb-watch` in a terminal tab".

**When to use:** Written by the installer script; loaded once via `launchctl bootstrap`.

**Critical launchd constraints (verified from `man launchd.plist` on this machine):**
- `ProgramArguments` requires **full absolute paths** — launchd does not inherit the user's shell PATH
- `KeepAlive: true` causes launchd to restart the process on any exit, **and** implies `RunAtLoad`
- Processes must not call `fork()`+`exit()` (i.e., must not daemonize themselves) — `sb-watch` is
  already correct (it loops in foreground)
- `WorkingDirectory` sets cwd before exec
- `EnvironmentVariables` injects key/value pairs into the process environment; use this to pass
  `PATH` so `uv` is found if the user's bin dir is non-standard
- `StandardOutPath` / `StandardErrorPath` redirect stdout/stderr to log files (important since there
  is no terminal)
- Plist filename convention: `<Label>.plist` — use `com.secondbrain.watch.plist`

**Load/unload (modern API, verified from `man launchctl` on this machine):**
```bash
# Load (idempotent — use bootstrap, not the deprecated 'load')
launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/com.secondbrain.watch.plist

# Unload
launchctl bootout gui/$(id -u)/com.secondbrain.watch

# Check status
launchctl print gui/$(id -u)/com.secondbrain.watch
```

**Plist template (generated via plistlib):**
```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
    "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>com.secondbrain.watch</string>
    <key>ProgramArguments</key>
    <array>
        <string>/Users/USERNAME/.local/bin/uv</string>
        <string>run</string>
        <string>--directory</string>
        <string>/Users/USERNAME/path/to/second-brain</string>
        <string>sb-watch</string>
    </array>
    <key>WorkingDirectory</key>
    <string>/Users/USERNAME/path/to/second-brain</string>
    <key>EnvironmentVariables</key>
    <dict>
        <key>PATH</key>
        <string>/Users/USERNAME/.local/bin:/usr/local/bin:/usr/bin:/bin</string>
        <key>HOME</key>
        <string>/Users/USERNAME</string>
    </dict>
    <key>KeepAlive</key>
    <true/>
    <key>StandardOutPath</key>
    <string>/Users/USERNAME/Library/Logs/second-brain-watch.log</string>
    <key>StandardErrorPath</key>
    <string>/Users/USERNAME/Library/Logs/second-brain-watch-error.log</string>
</dict>
</plist>
```

**Why `uv run --directory` rather than calling `sb-watch` directly after `uv tool install`?**
Two options for `ProgramArguments`:
1. `["/Users/USER/.local/bin/sb-watch"]` — works only after `uv tool install`; simpler
2. `["/Users/USER/.local/bin/uv", "run", "--directory", REPO, "sb-watch"]` — works directly from
   repo without requiring global tool install first

Prefer option 1 (direct shim path) when installer runs `uv tool install` first — fewer moving parts,
more explicit. The installer verifies the shim exists before writing the plist.

**Python plistlib generation:**
```python
# Source: plistlib stdlib docs + man launchd.plist (verified on this machine)
import plistlib
from pathlib import Path

uv_bin = str(Path.home() / ".local" / "bin" / "uv")
sb_watch_bin = str(Path.home() / ".local" / "bin" / "sb-watch")
log_dir = Path.home() / "Library" / "Logs"
log_dir.mkdir(parents=True, exist_ok=True)

plist = {
    "Label": "com.secondbrain.watch",
    "ProgramArguments": [sb_watch_bin],
    "WorkingDirectory": str(Path.home() / "SecondBrain"),
    "EnvironmentVariables": {
        "PATH": f"{Path.home()}/.local/bin:/usr/local/bin:/usr/bin:/bin",
        "HOME": str(Path.home()),
    },
    "KeepAlive": True,
    "StandardOutPath": str(log_dir / "second-brain-watch.log"),
    "StandardErrorPath": str(log_dir / "second-brain-watch-error.log"),
}

plist_path = Path.home() / "Library" / "LaunchAgents" / "com.secondbrain.watch.plist"
with open(plist_path, "wb") as f:
    plistlib.dump(plist, f)
```

### Pattern 3: Git Hook Installer

**What:** For each project repo the user wants monitored, run:
```
git config core.hooksPath /absolute/path/to/second-brain/.githooks
```
This points Git at the shared `.githooks/` directory already present in the repo, identical to how
the brain repo itself is configured.

**When to use:** On demand — the user passes one or more repo paths to the installer.

**Key facts:**
- The `.githooks/post-commit` shell script already exists and works correctly
- `core.hooksPath` is a per-repo local config setting — it does NOT require `--global`
- No hook files need to be copied; the single `.githooks/` directory serves all repos
- The installer should verify the target directory is a git repo before writing config
- Idempotent: running twice does not break anything

```python
# Source: git-config man page + project decision record [Phase 01-foundation]
import subprocess
from pathlib import Path

def install_hook(repo_path: Path, hooks_dir: Path) -> None:
    """Point repo_path's git hooks at hooks_dir."""
    result = subprocess.run(
        ["/usr/bin/git", "-C", str(repo_path), "rev-parse", "--git-dir"],
        capture_output=True, text=True,
    )
    if result.returncode != 0:
        raise ValueError(f"Not a git repo: {repo_path}")
    subprocess.run(
        ["/usr/bin/git", "-C", str(repo_path),
         "config", "core.hooksPath", str(hooks_dir)],
        check=True,
    )
    print(f"Installed hook: {repo_path}")
```

**Note:** Must use `/usr/bin/git -C <path>` not bare `git` — project CLAUDE.md requirement
(scm_breeze shell plugin breaks bare `git` on this machine).

### Anti-Patterns to Avoid

- **Symlinks to `sb-watch` in `/usr/local/bin`**: Fragile — venv Python path changes on `uv` upgrades. Use `uv tool install` shims instead.
- **`launchctl load` (deprecated)**: Prints warning on macOS 13+. Use `launchctl bootstrap gui/$(id -u)` instead.
- **Hardcoding `/usr/local/bin/uv`**: `uv` is at `~/.local/bin/uv` on this machine. Always resolve via `shutil.which("uv")` or `Path.home() / ".local/bin/uv"`.
- **Global `core.hooksPath` (`git config --global`)**: Would affect ALL repos including third-party — dangerous. Use per-repo config only.
- **Daemonizing `sb-watch` in Python**: `launchd` manages the process lifecycle; `sb-watch` must stay in foreground (already correct).
- **Writing plist as XML string template**: Use `plistlib.dump()` — avoids XML escaping bugs in paths with special characters.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---|---|---|---|
| Global CLI shims | Custom symlink/wrapper scripts | `uv tool install --editable` | uv manages isolated venv, handles PATH, upgradeable |
| Process autostart | Custom login item or cron job | launchd plist (LaunchAgent) | System-managed, restart-on-crash, log capture, macOS standard |
| Plist XML generation | f-string XML template | `plistlib.dump()` (stdlib) | Type-safe, handles special chars in paths |
| Multi-repo hook distribution | Copying hook files into each repo | `git config core.hooksPath` pointing at shared dir | Single source of truth, updates propagate automatically |

**Key insight:** All three problems have standard macOS/uv solutions that are more robust than hand-rolled equivalents. The installer script's job is wiring them together, not reimplementing them.

## Common Pitfalls

### Pitfall 1: launchd PATH is minimal
**What goes wrong:** `sb-watch` (or `uv`) not found when launched by launchd even though `which uv` works in terminal.
**Why it happens:** launchd launches with a minimal PATH (`/usr/bin:/bin:/usr/sbin:/sbin`), not the user's shell PATH.
**How to avoid:** Always set `EnvironmentVariables.PATH` in the plist to include `~/.local/bin`. Resolve the full path to `uv` and `sb-watch` at plist-generation time (`shutil.which("uv")`) and embed absolute paths in `ProgramArguments`.
**Warning signs:** `launchctl print gui/UID/com.secondbrain.watch` shows repeated exit code 1 or 127.

### Pitfall 2: uv tool install breaks on pyproject.toml changes
**What goes wrong:** After adding a new `[project.scripts]` entry, the old shims persist and the new one is missing.
**Why it happens:** `uv tool install --editable` does NOT auto-update shims when `pyproject.toml` changes.
**How to avoid:** Run `uv tool install --editable --force .` after any `pyproject.toml` entry point change. Document this in the installer and in comments.
**Warning signs:** `which sb-newcommand` returns nothing even though function exists.

### Pitfall 3: launchctl bootstrap vs load confusion
**What goes wrong:** Running `launchctl load ~/Library/LaunchAgents/com.secondbrain.watch.plist` works but prints deprecation warning; `launchctl bootstrap` fails with cryptic error.
**Why it happens:** Modern API requires domain target: `launchctl bootstrap gui/$(id -u) <plist>`.
**How to avoid:** Use `subprocess.run(["launchctl", "bootstrap", f"gui/{os.getuid()}", str(plist_path)])`.
**Warning signs:** "Unrecognized target specifier" error from launchctl.

### Pitfall 4: Double-loading plist
**What goes wrong:** Running the installer twice causes `launchctl bootstrap` to fail because the service is already loaded.
**Why it happens:** `bootstrap` errors if service already exists in the domain.
**How to avoid:** Before bootstrapping, check if loaded (`launchctl print gui/UID/com.secondbrain.watch`), bootout first if so. Or: check plist mtime — if plist unchanged, skip reloading.
**Warning signs:** Exit code 36 ("Service already loaded").

### Pitfall 5: Brain repo path baked into plist at install time
**What goes wrong:** User moves the second-brain repo; launchd keeps launching the old path.
**Why it happens:** Absolute paths are embedded in the plist at write time.
**How to avoid:** Document that the plist must be regenerated (re-run `install_native.py`) if the repo is moved. The installer should always derive the repo path from `__file__` (its own location relative to repo root).
**Warning signs:** `sb-watch` fails to start; check log at `~/Library/Logs/second-brain-watch-error.log`.

### Pitfall 6: Git bare `git` command
**What goes wrong:** Hook installer calls `subprocess.run(["git", "-C", ...])` and fails silently on this machine.
**Why it happens:** `scm_breeze` shell plugin breaks the bare `git` binary (documented in project CLAUDE.md).
**How to avoid:** Always use `/usr/bin/git -C <path>` in all subprocess calls to git.

## Code Examples

### Installer entry point (scripts/install_native.py skeleton)
```python
#!/usr/bin/env python3
"""One-command native macOS setup: global CLI + launchd watcher + git hook installer.

Usage:
    python scripts/install_native.py            # install all three
    python scripts/install_native.py --cli      # global CLI only
    python scripts/install_native.py --launchd  # launchd agent only
    python scripts/install_native.py --hooks /path/to/repo1 /path/to/repo2
"""
import argparse, os, plistlib, shutil, subprocess, sys
from pathlib import Path

REPO_ROOT = Path(__file__).resolve().parent.parent
HOOKS_DIR = REPO_ROOT / ".githooks"
PLIST_LABEL = "com.secondbrain.watch"
PLIST_PATH = Path.home() / "Library" / "LaunchAgents" / f"{PLIST_LABEL}.plist"
```

### Detecting uv bin path reliably
```python
# Source: verified on this machine — uv at ~/.local/bin/uv
def find_uv() -> Path:
    found = shutil.which("uv")
    if found:
        return Path(found).resolve()
    fallback = Path.home() / ".local" / "bin" / "uv"
    if fallback.exists():
        return fallback
    raise FileNotFoundError("uv not found. Install: https://docs.astral.sh/uv/")
```

### Idempotent launchd load
```python
# Source: man launchctl (verified on this machine)
def load_launchd_agent(plist_path: Path) -> None:
    uid = os.getuid()
    domain = f"gui/{uid}"
    label = f"{domain}/{PLIST_LABEL}"
    # Bootout first (ignore error if not loaded)
    subprocess.run(["launchctl", "bootout", label], capture_output=True)
    result = subprocess.run(
        ["launchctl", "bootstrap", domain, str(plist_path)],
        capture_output=True, text=True,
    )
    if result.returncode != 0:
        raise RuntimeError(f"launchctl bootstrap failed: {result.stderr}")
    print(f"launchd agent loaded: {PLIST_LABEL}")
```

## Open Questions

1. **`sb` shorthand CLI alias**
   - What we know: entry points are `sb-init`, `sb-capture`, etc. — multi-word, not a single `sb` command
   - What's unclear: does the user want a top-level `sb` dispatcher (e.g., `sb capture`, `sb search`) or is the existing `sb-*` namespace sufficient?
   - Recommendation: add a new `[project.scripts]` entry `sb = "engine.cli:main"` with a simple argparse dispatcher, only if the user wants it; otherwise skip

2. **DevContainer compatibility**
   - What we know: the system dual-boots between DevContainer (container paths) and host macOS (host paths). `engine/paths.py` auto-detects via `/workspace` existence.
   - What's unclear: should `sb-watch` launchd agent be skipped/no-op when run inside DevContainer?
   - Recommendation: `sb-watch` already uses `BRAIN_ROOT` detection; the launchd agent is host-only by definition (launchd doesn't run inside Docker), so no special handling needed.

3. **How many repos should the hook installer target?**
   - What we know: the user can pass arbitrary repo paths to the installer
   - What's unclear: should the installer scan common dirs (~/code, ~/projects) for git repos automatically?
   - Recommendation: explicit paths only (passed as args); auto-scanning is fragile and potentially slow on large home dirs.
