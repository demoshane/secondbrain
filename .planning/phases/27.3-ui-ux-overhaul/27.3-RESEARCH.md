## Summary

This phase replaces 1,677 lines of vanilla JS/HTML/CSS with a React 18 + Vite 5 + shadcn/ui + Tailwind CSS application. The Flask backend is completely untouched — it continues serving from `engine/gui/static/`, which becomes the Vite build output directory. pywebview wraps Flask and loads `/ui`; this mechanism is unchanged because Vite's production build emits a static `index.html` that Flask reads and injects `window.API_BASE` into.

The migration is a standard Vite + React + shadcn setup with one important production constraint: `build.base` must be `'./'` (relative assets) so Flask's `send_from_directory` serves assets correctly without a dedicated asset CDN. Development uses Vite's dev server with a proxy to forward `/notes`, `/search`, `/events`, etc. to Flask on port 37491.

The most complex pieces are: (1) the SSE `EventSource` lifecycle as a React context/hook, (2) the sidebar collapse state that persists to the server via `PUT /ui/prefs`, and (3) CodeMirror 6 integration as the markdown editor. All are well-supported by the chosen stack.

**Primary recommendation:** Scaffold with `npm create vite@latest frontend -- --template react` inside the repo root, configure `vite.config.ts` to build into `engine/gui/static/`, run `npx shadcn@latest init`, then migrate feature-by-feature starting with the shell layout and working inward.

### Pattern 1: Vite Build Output to Flask Static Dir
**What:** `vite.config.ts` sets `build.outDir` to `'../engine/gui/static'` and `base` to `'./'` so all asset URLs are relative.
**When to use:** Required for production; Flask's `gui_static()` route already serves any file under `_STATIC_DIR`.
**Example:**
```typescript
// Source: https://vite.dev/config/build-options
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'
import path from 'path'

export default defineConfig({
  plugins: [react()],
  base: './',                             // relative assets — required for Flask serving
  build: {
    outDir: '../engine/gui/static',       // overwrites current vanilla JS files
    emptyOutDir: true,                    // clear stale assets on each build
  },
  resolve: {
    alias: { '@': path.resolve(__dirname, './src') },
  },
  server: {
    port: 5173,
    proxy: {
      '/notes': 'http://127.0.0.1:37491',
      '/search': 'http://127.0.0.1:37491',
      '/events': 'http://127.0.0.1:37491',
      '/actions': 'http://127.0.0.1:37491',
```

### Pattern 2: API_BASE Injection Compatibility
**What:** Flask's `gui_shell()` injects `window.API_BASE` by string-replacing `</head>` in `index.html`. The React app must read this in production.
**When to use:** Production (pywebview) only. Dev server talks to Flask directly via Vite proxy, so `API_BASE` is unused.
**Example:**
```typescript
// src/lib/utils.ts
export const API = (window as any).API_BASE ?? 'http://127.0.0.1:37491'
```
This is identical to the current vanilla JS pattern and requires no Flask changes.

### Pattern 3: SSE Context with EventSource
**What:** A `SSEContext` holds the live EventSource and broadcasts note change events to all consumers.
**When to use:** Provides sidebar refresh + viewer update without prop drilling through the entire tree.
**Example:**
```typescript
// src/contexts/SSEContext.tsx
// Source: https://oneuptime.com/blog/post/2026-01-15-server-sent-events-sse-react/view
const SSEContext = createContext<{ connected: boolean }>({ connected: false })

export function SSEProvider({ children }: { children: React.ReactNode }) {
  const [connected, setConnected] = useState(false)
  const { loadNotes, openNote, currentPath } = useNoteContext()

  useEffect(() => {
    const src = new EventSource(`${API}/events`)
    src.onopen = () => {
      setConnected(true)
      // Reconnect refresh: reload notes when reconnecting after drop
    }
    src.onmessage = (e) => {
      const data = JSON.parse(e.data)
      loadNotes()
      if (currentPath && data.path === currentPath) openNote(currentPath)
    }
    src.onerror = () => setConnected(false)
    return () => src.close()
  }, [])   // empty deps — connect once, reconnect is EventSource native behavior

  return <SSEContext.Provider value={{ connected }}>{children}</SSEContext.Provider>
}
```

### Pattern 4: shadcn Dark Mode via prefers-color-scheme
**What:** `useEffect` on mount reads `matchMedia`, sets `dark` class on `<html>`, and listens for system theme changes.
**When to use:** Required; no manual toggle needed.
**Example:**
```typescript
// src/main.tsx or a ThemeProvider component
// Source: https://ui.shadcn.com/docs/dark-mode/vite
useEffect(() => {
  const mq = window.matchMedia('(prefers-color-scheme: dark)')
  const apply = (e: MediaQueryListEvent | MediaQueryList) => {
    document.documentElement.classList.toggle('dark', e.matches)
  }
  apply(mq)
  mq.addEventListener('change', apply)
  return () => mq.removeEventListener('change', apply)
}, [])
```
shadcn's CSS variables are defined under `:root` (light) and `.dark` — adding/removing the `dark` class on `<html>` is the entire mechanism.

### Pattern 5: CodeMirror 6 Markdown Editor
**What:** `@uiw/react-codemirror` wraps CodeMirror 6 as a controlled React component.
**When to use:** Note editor, replacing EasyMDE.
**Example:**
```typescript
// src/components/NoteEditor.tsx
// Source: https://uiwjs.github.io/react-codemirror/
import CodeMirror from '@uiw/react-codemirror'
import { markdown } from '@codemirror/lang-markdown'
import { oneDark } from '@codemirror/theme-one-dark'

export function NoteEditor({ value, onChange, isDark }: Props) {
  return (
    <CodeMirror
      value={value}
      extensions={[markdown()]}
      theme={isDark ? oneDark : 'light'}
      onChange={onChange}
      height="100%"
      basicSetup={{ lineNumbers: false, foldGutter: false }}
    />
  )
}
```

### Anti-Patterns to Avoid
- **Keeping `window.API_BASE` as a React module constant at module parse time:** It won't be injected yet. Read it lazily inside a function or `useEffect`.
- **Using `dangerouslySetInnerHTML` for markdown:** Use `react-markdown` instead.
- **Mounting the React app inside the existing `index.html`:** The entire `engine/gui/static/` content is replaced. Write a clean `frontend/index.html`.
- **Importing shadcn components from `shadcn/ui` directly:** Components live in `src/components/ui/` after `npx shadcn add`; they're local copies, not npm imports.
- **Forgetting `emptyOutDir: true`:** Without it, old `app.js`, `style.css`, `vendor/` files remain alongside the React build and may conflict.
- **Setting `base: '/'`:** This makes asset URLs absolute (`/assets/index-xxx.js`). Flask's `send_from_directory` will serve them, but it means every asset must be fetched relative to the domain root. `'./'` is safer and matches how Flask serves from `/ui/`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Markdown rendering | Manual `innerHTML` + marked.js | `react-markdown` + `remark-gfm` | XSS sanitization, React reconciliation, custom component mapping |
| Code editor | Textarea with tab handling | `@uiw/react-codemirror` | Undo/redo, syntax highlighting, resize, accessibility |
| Dark mode detection | Manual class toggling | shadcn's documented ThemeProvider pattern | Handles system change events, SSR-safe |
| CSS utility composition | String concatenation | `cn()` from `clsx` + `tailwind-merge` (installed by shadcn) | Deduplicates Tailwind classes, no specificity bugs |
| Icon set | Custom SVGs | `lucide-react` (bundled with shadcn) | Tree-shakeable, consistent sizing |
| Dialog/Modal | `display:none` toggles | shadcn `Dialog` component | Focus trap, keyboard dismiss, accessibility |
| Select/Dropdown | Native `<select>` | shadcn `Select` component | Consistent styling across OS |

**Key insight:** shadcn/ui provides every primitive needed (Button, Input, Select, Dialog, Table, Badge, ScrollArea) — use them for all UI elements rather than raw HTML elements styled with Tailwind.

## Common Pitfalls

### Pitfall 1: API_BASE not available at module parse time
**What goes wrong:** `const API = window.API_BASE ?? 'http://127.0.0.1:37491'` works in vanilla JS but in a Vite module, `window.API_BASE` is injected via string-replace into `index.html` before the `<script>` tag. If the value is read before the script tag runs (e.g., at module level in a non-entry module), it may be `undefined`.
**Why it happens:** Vite bundles all modules before script execution; `window.API_BASE` is injected by Flask's template engine at request time, after bundling.
**How to avoid:** Define `API` in a single `src/lib/utils.ts` that's imported lazily, or read `window.API_BASE` inside the first `useEffect` (guaranteed post-mount = post-injection).
**Warning signs:** API calls going to `undefined/notes` or `undefined/events`.

### Pitfall 2: `emptyOutDir` deletes itself when `outDir` is outside project root
**What goes wrong:** Vite warns "outDir is outside root" and may refuse `emptyOutDir` by default.
**Why it happens:** Vite's default is to only auto-empty if `outDir` is inside the project root; `../engine/gui/static` is outside `frontend/`.
**How to avoid:** Set `build.emptyOutDir: true` explicitly in `vite.config.ts` — this overrides the safety check.
**Warning signs:** Old `app.js` or `style.css` still present after `npm run build`.

### Pitfall 3: shadcn path alias `@` not configured
**What goes wrong:** `npx shadcn@latest init` generates components that import from `@/lib/utils`. If `@` is not resolved in `vite.config.ts` and `tsconfig.json`, build fails.
**Why it happens:** shadcn assumes the `@` alias is set up; it doesn't configure it automatically for Vite.
**How to avoid:** Add `resolve.alias: { '@': path.resolve(__dirname, './src') }` in `vite.config.ts` before running `shadcn init`. Also add `"paths": { "@/*": ["./src/*"] }` in `tsconfig.json`.
**Warning signs:** `Cannot find module '@/lib/utils'` at build time.

### Pitfall 4: Vite dev proxy doesn't forward `/ui/prefs`
**What goes wrong:** The sidebar collapse state is persisted to `PUT /ui/prefs`. If this path isn't proxied in dev, requests 404.
**Why it happens:** Developers often proxy `/notes`, `/search`, etc. but forget less obvious paths.
**How to avoid:** Proxy all `/ui/*` prefixed routes in addition to the data routes.
**Warning signs:** Collapse state not persisting in Vite dev mode.

### Pitfall 5: SSE EventSource URL mismatch in dev vs production
**What goes wrong:** In dev, `new EventSource('/events')` works because Vite proxies it. In production (pywebview), it resolves to Flask's origin. But if `API` is defined as an absolute URL, `new EventSource(`${API}/events`)` works in both.
**Why it happens:** Inconsistent handling of absolute vs relative URLs.
**How to avoid:** Always construct EventSource URL as `new EventSource(\`${API}/events\`)` using the same `API` constant.

### Pitfall 6: `react-markdown` version 9 breaking change
**What goes wrong:** `react-markdown` v9 dropped CommonJS support and uses ESM-only. Import must be `import Markdown from 'react-markdown'` not `require`.
**Why it happens:** The package moved to pure ESM in v9.
**How to avoid:** Vite handles ESM natively — this is not an issue with Vite. Just use the named default import.

### Pitfall 7: EasyMDE CSS leaking
**What goes wrong:** Old `vendor/easymde.min.css` left in `engine/gui/static/` (if `emptyOutDir` is not set) could be loaded by any stale reference.
**Why it happens:** `emptyOutDir: true` not set; old vendor directory survives build.
**How to avoid:** Confirm `emptyOutDir: true` and verify `vendor/` directory is gone after first `npm run build`.

### NoteContext skeleton
```typescript
// src/contexts/NoteContext.tsx
import { createContext, useContext, useState, useCallback } from 'react'
import { API } from '@/lib/utils'
import type { Note } from '@/types'

interface NoteContextValue {
  notes: Note[]
  currentPath: string | null
  currentNote: Note | null
  loadNotes: () => Promise<void>
  openNote: (path: string) => Promise<void>
  isDirty: boolean
  setIsDirty: (v: boolean) => void
}

const NoteContext = createContext<NoteContextValue>(null!)
export const useNoteContext = () => useContext(NoteContext)

export function NoteProvider({ children }: { children: React.ReactNode }) {
  const [notes, setNotes] = useState<Note[]>([])
  const [currentPath, setCurrentPath] = useState<string | null>(null)
  const [currentNote, setCurrentNote] = useState<Note | null>(null)
  const [isDirty, setIsDirty] = useState(false)

  const loadNotes = useCallback(async () => {
    const res = await fetch(`${API}/notes`)
    const { notes } = await res.json()
    setNotes(notes)
  }, [])

  const openNote = useCallback(async (path: string) => {
    const encoded = encodeURIComponent(path)
    const res = await fetch(`${API}/notes/${encoded}`)
    const note = await res.json()
    setCurrentPath(path)
    setCurrentNote(note)
  }, [])

  return (
    <NoteContext.Provider value={{ notes, currentPath, currentNote, loadNotes, openNote, isDirty, setIsDirty }}>
      {children}
    </NoteContext.Provider>
  )
}
```

## Open Questions

1. **`frontend/` directory location — repo root vs inside `engine/`**
   - What we know: Vite project needs `package.json`; `engine/` is Python-only today
   - What's unclear: Whether `frontend/` at repo root vs `engine/gui/frontend/` is cleaner
   - Recommendation: Place `frontend/` at repo root, sibling to `engine/`. Build output goes `../engine/gui/static`. This keeps Python and JS tooling cleanly separated.

2. **`package.json` in repo root — interference with Python tooling**
   - What we know: Repo has no `package.json` today; pytest, uv, pyproject.toml are at root
   - What's unclear: Whether a `frontend/` subdirectory is preferable to avoid `node_modules/` at root
   - Recommendation: Keep `frontend/` as a subdirectory. Avoids `node_modules/` at repo root, keeps CI simple.

3. **Playwright tests after migration — selectors will break**
   - What we know: `tests/test_gui.py` uses `#note-list li[data-path]`, `#viewer`, `#sidebar-loading` etc.
   - What's unclear: Whether to update Playwright selectors in this phase or leave them as known-broken
   - Recommendation: Update Playwright selectors in this phase. The tests directly validate feature parity (success criteria #2). Use `data-testid` attributes on React components to make selectors stable.

4. **`window.API_BASE` injection — Vite's `index.html` processing**
   - What we know: Flask string-replaces `</head>` in `index.html` to inject `<script>window.API_BASE = "...";</script>`
   - What's unclear: Vite may add its own content-hash scripts before `</head>` in development, but in production the built `index.html` has a literal `</head>` tag that Flask can replace
   - Recommendation: This is LOW risk. Vite's built `index.html` contains a plain `</head>` — Flask's injection works unchanged. Verify once after first `npm run build`.
