# Phase 27.3: React + Shadcn/UI Migration - Context

**Gathered:** 2026-03-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Migrate the entire GUI frontend from vanilla JS (index.html + app.js + style.css) to React + Vite + shadcn/ui + Tailwind CSS, with full feature parity and automatic dark mode. Backend (Flask API, engine/) is untouched. This is a big-bang frontend rewrite — no legacy vanilla JS survives.

</domain>

<decisions>
## Implementation Decisions

### Build tooling
- **Vite** as the React build tool — outputs static files that Flask serves
- `npm run build` should write to `engine/gui/static/` (or an equivalent path Flask serves)
- Development: `vite dev` proxies API calls to Flask

### Migration strategy
- **Big bang rewrite** — replace index.html + app.js + style.css entirely with a React app
- No incremental mounting or vanilla JS compatibility shims
- All existing features must be reimplemented; nothing deferred to a later phase

### Design system
- **shadcn/ui** — installed via `npx shadcn@latest add`; components live in `src/components/ui/`
- **Tailwind CSS** — utility-first styling, Tailwind config generated by shadcn init
- shadcn's color token system (CSS custom properties on `:root` / `.dark`) replaces the current hard-coded hex values

### Dark mode
- **Auto via `prefers-color-scheme`** — no manual toggle needed
- shadcn's class strategy (add `dark` class to `<html>`) driven by a `useEffect` that reads `window.matchMedia`

### Feature parity (all required — no deferrals)
- Topbar: search input, search mode selector, New Note, File Upload, Batch Capture, SSE status dot
- Tab bar: Notes / Action Items tabs
- Sidebar: grouped folder/type hierarchy, tag filter banner, collapse/expand
- Note viewer: rendered markdown (marked.js or react-markdown)
- Note editor: **CodeMirror 6** (not EasyMDE) — lighter, has a React wrapper, closer to EasyMDE feel
- Intelligence panel (right column): recap, connections, digest
- Actions page: table, filters, assignee picker, done checkbox
- File upload modal + attachment list below note body
- Batch capture button + modal
- SSE live refresh for sidebar and viewer updates

### State management
- **React Context + hooks** — no external state library
- Likely contexts: `NoteContext` (active note, list), `SearchContext`, `UIContext` (tab, sidebar state), `SSEContext` (live event stream)
- Add Zustand only if Context causes prop-drilling pain during implementation

### Claude's Discretion
- Exact component file/folder structure within `src/`
- Whether to use `react-markdown` or port marked.js usage
- Routing approach (no router needed — single-page app with tab state)
- How to handle EasyMDE's CSS (replace entirely with CodeMirror theme)
- Test setup: Vitest + React Testing Library alongside existing pytest backend tests

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `engine/api.py`: All Flask REST endpoints remain unchanged — React fetches the same `/notes`, `/search`, `/actions`, `/capture` etc.
- `engine/gui/static/vendor/marked.min.js`: Can be replaced by `react-markdown` npm package
- `engine/gui/static/vendor/easymde.min.js`: Replaced by CodeMirror 6 React wrapper (`@uiw/react-codemirror`)

### Established Patterns
- Backend is authoritative — all data transformation stays server-side
- SSE endpoint at `/events` drives live refresh; React will consume via `useEffect` + `EventSource`
- Three-panel layout (sidebar / center / intelligence) is the core shell — reproduce in React flexbox/grid
- API contract: REST JSON, no GraphQL, no websockets (SSE only for push)

### Integration Points
- Flask serves the built React app from `engine/gui/static/` — `index.html` entry point stays there
- pywebview wraps the Flask server; the React bundle must be a static build (no Node server in production)
- All existing pytest tests cover backend only — they remain unaffected by the frontend rewrite
- Frontend tests (if added): Vitest + React Testing Library, run separately from pytest

</code_context>
