<objective>
Create note_tags and note_people junction tables with proper indexes, migrate existing JSON data into them, update capture/update paths to dual-write, and switch read queries to use the indexed junction tables.

Purpose: Tags and people stored as JSON TEXT cause full-table scans on filter. Junction tables with indexes give O(1) lookups. idx_notes_people on the JSON blob is useless — drop it.
Output: Two junction tables, populated by migration, dual-written on capture/update, read by queries.
</objective>

  <action>
    In engine/db.py, add two migration functions:

    `migrate_add_note_tags_table(conn)`:
    - CREATE TABLE IF NOT EXISTS note_tags with FK ON DELETE CASCADE to notes(path)
    - CREATE INDEX IF NOT EXISTS on tag and note_path columns
    - SELECT path, tags FROM notes WHERE tags != '[]'
    - For each row, json.loads(tags) and INSERT OR IGNORE into note_tags
    - Log count via logging.info()

    `migrate_add_note_people_table(conn)`:
    - CREATE TABLE IF NOT EXISTS note_people with FK ON DELETE CASCADE to notes(path)
    - CREATE INDEX IF NOT EXISTS on person and note_path columns
    - DROP INDEX IF EXISTS idx_notes_people (the useless JSON text index)
    - SELECT path, people FROM notes WHERE people IS NOT NULL AND people != '[]'
    - For each row, json.loads(people) and INSERT OR IGNORE into note_people
    - Log count via logging.info()

    Register both in init_schema() AFTER migrate_paths_to_relative (per locked migration order).

    Note: Junction table paths must use the NEW relative format (since path migration runs first). The FK references notes(path), so the paths must match.

    Tests: insert notes with JSON tags/people, call migration, verify junction tables populated, verify idempotency, verify old index dropped.
