<decisions>
## Implementation Decisions

### Out-of-scope bug handling
- When a test agent discovers a broken existing feature, it logs it as a GSD todo (`/gsd:add-todo`) and does NOT fix it inline
- The discovered bug is written as a test marked `@pytest.mark.xfail(strict=False, reason='known bug: ... — see GSD todo #X')`
- If the bug blocks the test agent from making progress (e.g., a core feature is completely broken), execution pauses and a separate focused fix plan is created before testing resumes
- This replaces the current implicit behavior where agents fix adjacent issues and cause cascading failures

### Continuous quality gate
- Playwright suite must pass (no non-xfail failures) before any phase is marked complete
- This is enforced as part of `/gsd:verify-work` / phase completion checks
- Every phase that adds a new feature must include at least one Playwright test for that feature in the same plan (test-alongside rule)

### Coverage target: Smoke + data
- Every tab/page must have: (1) the page container renders, (2) at least one key data element shows real content (not just element exists)
- NOT full interaction testing — that's out of scope for this phase

### People/person type regression test (explicit)
- Write a dedicated regression test that seeds a note with `type='people'`, verifies it appears on the People page with correct content
- The test must confirm that `type='person'` notes are not mixed in incorrectly
- Guards exactly the regression where alice smith appeared as "old content" / broken type confusion

### Orphan detection logic tests
- Write tests that define correct orphan behavior: what IS an orphan (no backlinks, no parent folder), what is NOT an orphan (has at least one link)
- If current orphan logic fails these tests → xfail + GSD todo; fix comes in a follow-up plan
- Tests establish the spec, even if current implementation doesn't pass them yet

### Right panel coverage
- Tests verify: right panel renders, backlinks section is present, people section is present (existence smoke)
- Additionally: seed a note that mentions "Test Person", verify Test Person appears in the right panel people section (data correctness for the people mentions regression class)

### Test fixtures — isolated test brain
- Tests must run against a separate isolated test brain directory (temp folder), never touching ~/SecondBrain
- Each note type gets a persistent QA fixture note: "Test Person", "Test Meeting", "Test Project", "Test Note" etc. — seeded once per test session, never the real brain
- These fixture notes contain all the features of their type (body, backlinks, action items etc.) so tests don't need to create/destroy content repeatedly
- Fixture notes cannot be deleted by tests (test infrastructure enforcement)

### Pre-flight smoke test
- Lives in `tests/test_preflight.py` — separate from test_gui.py, runs first
- Checks: /health, /brain-health, /notes, /people, /meetings, /projects all return 200 AND /ui renders the GUI
- If pre-flight fails: all remaining tests are skipped with a clear message identifying which endpoint failed
- This catches the /brain-health 500 regression class before the full suite runs

### Claude's Discretion
- Exact xfail reason string formatting
- Order of tests within test files
- Whether to use conftest.py session fixture or inline setup for isolated test brain

</decisions>

## Existing Code Insights

### Reusable Assets
- `tests/test_gui.py`: existing 17KB Playwright suite — reference for patterns (page fixture, live_server_url, gui_brain, seed_note_fn)
- `gui_brain` fixture in conftest.py: temp brain dir — extend to seed QA fixture notes
- `seed_note_fn`: existing helper that creates notes via API — reuse for fixture seeding

### Established Patterns
- Tests use `data-testid` selectors throughout — maintain this pattern for all new tests
- API-based seeding (POST /notes) rather than direct DB writes — consistent, tests the capture path
- `uv run pytest tests/test_gui.py -v` is the run command

### Integration Points
- `tests/conftest.py` — where gui_brain and seed_note_fn live; extend here for isolated QA brain and fixture notes
- `/brain-health` endpoint — covered by pre-flight (was 500ing before; regression guard needed)
- RightPanel.tsx — no data-testid attributes exist yet; may need to add them to the frontend as part of this phase

### Coverage gaps (existing tests DO NOT cover)
- RightPanel (backlinks section, people mentions section) — zero tests
- Meetings tab content (only navigation exists, no data checks)
- Notes sidebar — no "notes show real data" check, only incidental coverage
- Orphan detection logic — only UI visibility tested, not correctness
- Pre-flight — no such test exists today
