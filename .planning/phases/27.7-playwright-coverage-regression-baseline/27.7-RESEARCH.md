## Summary

The project has a mature Playwright suite (`tests/test_gui.py`, ~380 lines) using pytest-playwright with session-scoped Flask daemon-thread server and `gui_brain` temp brain fixture. The suite currently covers: markdown rendering, scroll, SSE live refresh, delete flow, tag editing/filtering, collapsible sections, path traversal guard, and tab navigation for People/Meetings/Projects/Intelligence pages.

**Critical gaps identified:** (1) RightPanel has zero Playwright tests despite having `data-testid="right-panel"`, `data-testid="backlink-item"`, and `data-testid="people-badge"` already in the source. (2) Tab tests navigate to pages but do not assert real data content — they pass even on empty DB. (3) No pre-flight test exists. (4) Orphan detection logic has only existence tests, not correctness tests. (5) The `people` vs `person` type confusion regression (alice smith bug) has no guard.

**Primary recommendation:** Extend existing test infrastructure without replacing it. Add `tests/test_preflight.py` as a new file, add new test functions to `tests/test_gui.py` for right panel and data correctness, add orphan behavior spec tests to `tests/test_brain_health.py`.

### Pattern 1: Extending gui_brain for QA Fixture Notes

- A `type='people'` note (distinct from `type='person'`) to test the people/person regression
- A "Test Mention Note" whose body contains "Test Person" to test right panel people detection

### Pattern 2: Pre-flight Smoke Test

Pre-flight must run before any Playwright tests. Using a separate file `test_preflight.py` gives pytest natural alphabetical ordering (p < t, but use `pytest-ordering` or just note that `--co` order may vary). The reliable pattern is a session-scoped autouse fixture or simply running the file explicitly first.

The simplest enforcement: use `pytest.ini` or `pyproject.toml` `testpaths` + file naming, OR use a conftest session-scoped autouse fixture that checks endpoints before tests run.

**Recommended approach:** Place pre-flight as a standalone test file. Document in RESEARCH that the Playwright tests import `live_server_url` which depends on `gui_brain` — the pre-flight tests should share the same `live_server_url` fixture so they run against the same server instance.

```python
# tests/test_preflight.py
import pytest
import requests

ENDPOINTS = ["/health", "/brain-health", "/notes", "/people", "/meetings", "/projects"]

@pytest.fixture(scope="session", autouse=False)
def preflight_check(live_server_url):
    """Verify all API endpoints return 200 before Playwright tests run."""
    failed = []
    for ep in ENDPOINTS:
        try:
            r = requests.get(f"{live_server_url}{ep}", timeout=5)
            if r.status_code != 200:
                failed.append(f"{ep} returned {r.status_code}")
        except Exception as e:
            failed.append(f"{ep} raised {e}")
    if failed:
        pytest.skip(
            f"Pre-flight failed: {', '.join(failed)} — fix before running full suite"
        )

def test_preflight_health(live_server_url):
    r = requests.get(f"{live_server_url}/health", timeout=5)
    assert r.status_code == 200

def test_preflight_brain_health(live_server_url):
    r = requests.get(f"{live_server_url}/brain-health", timeout=5)
    assert r.status_code == 200, f"Pre-flight failed: /brain-health returned {r.status_code}"

def test_preflight_notes_list(live_server_url):
    r = requests.get(f"{live_server_url}/notes", timeout=5)
    assert r.status_code == 200

def test_preflight_ui_renders(page, live_server_url):
    page.goto("/ui")
    page.locator('[data-testid="tab-bar"]').wait_for(state="visible", timeout=5000)
```

### Pattern 3: Right Panel Tests

Right panel only renders when a note is open (`currentPath` in NoteContext). The `/notes/<path>/meta` endpoint drives both backlinks and people sections. Key findings:

- `data-testid="right-panel"` — the ScrollArea container (always rendered)
- `data-testid="backlink-item"` — only rendered when `backlinks.length > 0`
- `data-testid="people-badge"` — only rendered when `people.length > 0`
- People detection uses body-mention fallback (LEARNINGS.md: "meta endpoint must detect people by body-mention") — so seeding a note whose body contains a person's title is sufficient to get people badges without needing the `people` frontmatter column populated

```python
def test_right_panel_renders(page, live_server_url, gui_brain, seed_note_fn):
    """Right panel container is visible when a note is open."""
    seed_note_fn(gui_brain, "Panel Test Note", "some body content")
    page.goto("/ui")
    page.locator('[data-testid="note-item"]').first.wait_for(state="visible", timeout=5000)
    page.locator('[data-testid="note-item"]', has_text="Panel Test Note").first.click()
    page.locator('[data-testid="note-viewer"]').wait_for(state="visible", timeout=5000)
    page.locator('[data-testid="right-panel"]').wait_for(state="visible", timeout=3000)

def test_right_panel_people_mention(page, live_server_url, gui_brain):
    """Opening a note that mentions Test Person shows Test Person in right panel people section."""
    page.goto("/ui")
    page.locator('[data-testid="note-item"]').first.wait_for(state="visible", timeout=5000)
    # Test Mention Note is seeded in gui_brain with "Test Person" in body
    page.locator('[data-testid="note-item"]', has_text="Test Mention Note").first.click()
    page.locator('[data-testid="note-viewer"]').wait_for(state="visible", timeout=5000)
    page.locator('[data-testid="right-panel"]').wait_for(state="visible", timeout=3000)
    # People badge for Test Person must appear
    people_badge = page.locator('[data-testid="people-badge"]', has_text="Test Person")
    people_badge.wait_for(state="visible", timeout=5000)
```

**Gotcha:** The `/notes/<path>/meta` endpoint performs body-mention detection by querying all `type IN ('person', 'people')` notes. The `Test Person` fixture note uses `type='person'`. The API must be running against the same `gui_brain` DB. This works because `live_server_url` depends on `gui_brain`.

### Anti-Patterns to Avoid

- **Writing tests that only check element existence:** Every tab test must assert at least one piece of real content (a note title, a score value, a row with seeded data). Existence-only tests pass on broken pages.
- **Creating notes in gui_brain from within individual tests:** The fixture is session-scoped. Individual tests should use `seed_note_fn` for ephemeral notes or rely on the session fixtures for persistent QA data. Do not delete session fixtures mid-suite.
- **Using bare `live_server_url` without `gui_brain`:** `live_server_url` depends on `gui_brain` for DB/path setup — listing `gui_brain` as a parameter in tests that need seeded data also ensures the fixture is fully initialized.
- **Importing from engine inside test functions instead of using the API:** Use `seed_note_fn` (POST /notes equivalent via direct DB) rather than importing `capture_note` directly — keeps tests consistent with existing patterns.

## Common Pitfalls

### Pitfall 1: Right Panel People Section Empty
**What goes wrong:** Test opens a note, checks for `[data-testid="people-badge"]`, finds nothing.
**Why it happens:** The `/notes/<path>/meta` endpoint does body-mention detection — it queries all `type IN ('person', 'people')` notes and checks if their title appears case-insensitively in the note body. If the person note is not in the DB at test time (race condition or fixture ordering), no badges appear.
**How to avoid:** The mention-target note (Test Mention Note) and the person note (Test Person) must both be seeded in `gui_brain` before the Flask server starts. Both are session-scoped seeds — verify they appear before the `return brain` line in the fixture.
**Warning signs:** Badge count is 0; `/notes/<path>/meta` returns `people: []`; check that `type='person'` note is in the isolated DB.

### Pitfall 2: gui_brain Session Fixture Ordering
**What goes wrong:** Tests fail because seeded data is absent, even though the fixture appears correct.
**Why it happens:** `gui_brain` is `scope="session"` — it runs once. If a test requests `seed_note_fn` but NOT `gui_brain` explicitly, the session fixture may not have run its full setup before the Flask server starts.
**How to avoid:** The `live_server_url` fixture already depends on `gui_brain`. Any test using `live_server_url` gets `gui_brain` initialized. Only tests that bypass `live_server_url` need to list `gui_brain` explicitly.

### Pitfall 3: Pre-flight /brain-health 500
**What goes wrong:** `/brain-health` returns 500 in the test environment.
**Why it happens:** Historical bug (LEARNINGS.md): `compute_health_score()` called with wrong kwarg `empty=` instead of `orphans=`. Fixed. New risk: `BRAIN_PATH` not set when Flask starts, causing `get_orphan_notes()` to use wrong DB path.
**How to avoid:** Pre-flight test must run after `live_server_url` fixture (which depends on `gui_brain` which sets `BRAIN_PATH`). The pre-flight test for `/brain-health` is the regression guard for this entire class of startup errors.
**Warning signs:** `/brain-health` returns 500 in pre-flight; check `BRAIN_PATH` env var and both `DB_PATH` module attributes.

### Pitfall 4: people vs person Type Confusion
**What goes wrong:** People page shows wrong notes or misses some.
**Why it happens:** The `list_people()` API endpoint must query `WHERE type IN ('person', 'people')` (both). Historical regression: the alice smith bug where `type='people'` plural was used for group-type notes but the page only queried `type='person'` singular.
**How to avoid:** Seed both `type='person'` (Test Person) and `type='people'` (Test Group) in `gui_brain`. The regression test must verify both appear on the People page.
**Warning signs:** Test Group (type='people') absent from people page table.

### Pitfall 5: Playwright Selector Timing for Right Panel
**What goes wrong:** `wait_for('[data-testid="people-badge"]')` times out.
**Why it happens:** The right panel makes a fetch to `/notes/<path>/meta` after the note opens. This is async — the panel renders empty first, then populates. Default Playwright auto-wait is 30s but `wait_for` default timeout may need adjustment.
**How to avoid:** Use `.wait_for(state="visible", timeout=5000)` on the badge locator after clicking a note. The fetch + render takes <500ms in the test server.

### Pitfall 6: Frontend Build Required for data-testid Changes
**What goes wrong:** New `data-testid` attributes added to React components are not found by Playwright.
**Why it happens:** Playwright tests against the built static bundle in `engine/gui/static/`, not the source TSX. See LEARNINGS.md: "any plan touching `frontend/src/**` must include the deploy pipeline."
**How to avoid:** RightPanel.tsx already has `data-testid="right-panel"`, `"backlink-item"`, `"people-badge"` — no new frontend changes required for the right panel tests. If any new testid is needed, the plan must include build + reinstall steps.
