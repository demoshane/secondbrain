## Summary

Phase 27.6 adds a dedicated Projects page following the identical pattern established by Phase 27.4 (People) and Phase 27.5 (Meetings). The backend needs two new Flask endpoints in `engine/api.py` that query `notes WHERE type='projects'` (note: the actual type value used in existing notes is `'projects'`, not `'project'`). The frontend needs a `ProjectsPage.tsx` component, a new `'projects'` entry in the `View` union, a tab in `TabBar.tsx`, and a routing branch in `App.tsx`.

The most important discovery from inspecting real project notes is that the `type` field in YAML frontmatter is `'projects'` (plural), not `'project'` (singular). This contrasts with `'person'` (singular) for people and `'meeting'` (singular) for meetings. The API query must use `WHERE n.type = 'projects'` to match existing notes.

Project notes do not have a structured `status` field in the database schema — status information is embedded in the note body as free text. There is no `status` column in the `notes` table. The detail view should render the full note body (which captures status naturally) rather than trying to extract a structured status field.

**Primary recommendation:** Follow the MeetingsPage pattern verbatim, swapping meeting terminology for project terminology. Use `type='projects'` in the SQL query. Display project body via react-markdown in the detail panel. Skip structured status — show full body instead.

### Pattern 1: Backend List Endpoint (from api.py lines 249-259 and 262-285)

**What:** SQL query with correlated subquery for open_actions count, row_factory=sqlite3.Row, JSON response.

**Critical note:** `type='projects'` (plural) — verified from real project notes in `~/SecondBrain/projects/`.

```python
# Source: engine/api.py lines 249-285 (list_people + list_meetings pattern)
@app.get("/projects")
def list_projects():
    conn = get_connection()
    conn.row_factory = sqlite3.Row
    rows = conn.execute(
        "SELECT n.path, n.title, substr(n.updated_at,1,10) AS updated_at, "
        "  (SELECT COUNT(*) FROM action_items a WHERE a.note_path=n.path AND a.done=0) AS open_actions "
        "FROM notes n WHERE n.type = 'projects' ORDER BY n.updated_at DESC"
    ).fetchall()
    conn.close()
    return jsonify({"projects": [dict(r) for r in rows]})
```

**Key decisions:**
- Order by `updated_at DESC` (most recently touched project first — more useful than title-alpha)
- `open_actions` counts by `note_path` (same as meetings), NOT by `assignee_path` (that's for people)
- No `people` column JSON parsing needed for the list — project notes have `people: []` but the list view doesn't need it

### Pattern 2: Backend Detail Endpoint (from api.py lines 288-319)

```python
# Source: engine/api.py lines 288-319 (get_meeting pattern)
@app.get("/projects/<path:note_path>")
def get_project(note_path):
    try:
        abs_path, _brain_root = _resolve_note_path(note_path)
    except ValueError:
        return jsonify({"error": "forbidden"}), 403
    conn = get_connection()
    conn.row_factory = sqlite3.Row
    row = conn.execute(
        "SELECT path, title, body, substr(updated_at,1,10) AS updated_at FROM notes WHERE path=?",
        (str(abs_path),)
    ).fetchone()
    if row is None:
        conn.close()
        return jsonify({"error": "not found"}), 404
    open_actions = conn.execute(
        "SELECT COUNT(*) FROM action_items WHERE note_path=? AND done=0",
        (str(abs_path),)
    ).fetchone()[0]
    conn.close()
    return jsonify({
        "path": row["path"],
        "title": row["title"] or "",
        "body": row["body"] or "",
        "updated_at": row["updated_at"] or "",
        "open_actions": open_actions,
    })
```

### Anti-Patterns to Avoid

- **Using `type='project'` (singular):** Real notes use `type='projects'` (plural). Wrong type = empty list always returns.
- **Adding a `status` DB column:** There is no status column in the schema; status is in the note body. Don't add a migration for this.
- **Filtering by `assignee_path`:** That's the people pattern. Projects use `note_path` for action items (same as meetings).
- **Using `page.wait_for_selector()` with `has_text`:** Established rule from [Phase 24] — use `page.locator(sel, has_text=).first.wait_for()` instead.
- **Committing App.tsx before ProjectsPage.tsx exists:** Both files must be committed together since App.tsx imports ProjectsPage (same rule as [Phase 27.5-meetings-page]).

## Common Pitfalls

### Pitfall 1: Wrong `type` value in SQL query
**What goes wrong:** Query returns empty list; projects page always shows "No projects found".
**Why it happens:** Assuming `type='project'` (singular) by analogy with `type='person'`, but project notes captured via `sb-capture` use `type='projects'` (plural).
**How to avoid:** Use `WHERE n.type = 'projects'` — verified directly from `~/SecondBrain/projects/2026-03-14-test-project.md` frontmatter (`type: projects`).
**Warning signs:** Unit test with seeded row passes but live GUI shows empty list.

### Pitfall 2: Committing App.tsx before ProjectsPage.tsx exists
**What goes wrong:** Build fails — `import { ProjectsPage } from './components/ProjectsPage'` resolves to nothing.
**Why it happens:** App.tsx import is added in Task 1 but the component file is Task 2.
**How to avoid:** Commit Tasks 1 and 2 together (same pattern as [Phase 27.5-meetings-page] decision: "Tasks 1+2 committed together").

### Pitfall 3: Using `page.goto(live_server_url)` without `/ui` suffix
**What goes wrong:** Playwright navigates to the Flask root, not the React app.
**Why it happens:** The React app is served at `/ui` by Flask.
**How to avoid:** Use `page.goto(live_server_url + '/ui')` — confirmed from [Phase 27.5-03]: "page.goto('/ui') used in new tests — not bare live_server_url".

Actually re-checking: [Phase 27.5-03] states `page.goto(live_server_url)` without `/ui`. The live_server_url fixture returns the base URL and the test uses `page.goto(live_server_url)`. Confirm by checking existing passing tests before writing new ones.

### Pitfall 4: Missing `conn.row_factory = sqlite3.Row` in backend endpoint
**What goes wrong:** `dict(r)` on a plain tuple raises a TypeError.
**Why it happens:** Forgetting to set row_factory before query.
**How to avoid:** Always add `conn.row_factory = sqlite3.Row` immediately after `conn = get_connection()` — exactly as in all existing endpoints.

### Pitfall 5: test fixture `people` column omitted from INSERT
**What goes wrong:** INSERT fails on NOT NULL constraint or columns don't match if `people` column is listed.
**Why it happens:** The notes table has `people TEXT NOT NULL DEFAULT '[]'` added via migration.
**How to avoid:** Project notes don't have meaningful people data — omit `people` from INSERT or pass `'[]'`; using `INSERT OR REPLACE INTO notes (path, title, type, body, tags, created_at, updated_at)` (without people) works because the column has a DEFAULT.

## Open Questions

1. **Should the Projects page show a `status` field separately?**
   - What we know: Real project notes do not have a structured `status` column in SQLite. The body text contains status descriptions inline (e.g., "What is the current status? Test").
   - What's unclear: The phase goal says "shows project notes with their status" — this likely means showing the note body, which contains status as free text.
   - Recommendation: Show full note body in the Note section. Do not add a `status` DB column — that would require a migration and changes to the capture path, which is out of scope for this phase.

2. **Should related notes be shown in a separate section?**
   - What we know: The phase goal says "related notes". The `relationships` table exists. The `/notes/<enc>/meta` backlinks endpoint returns related notes via the relationships table.
   - What's unclear: Whether "related notes" means backlinks (notes that link TO this project) or relationships (explicit links in the relationships table), or both.
   - Recommendation: Use backlinks from `GET /notes/<enc>/meta` — same as MeetingsPage and PeoplePage. This is what the existing infrastructure supports and what users would expect. Label the section "Related Notes" instead of "Backlinks" for clearer UX.
