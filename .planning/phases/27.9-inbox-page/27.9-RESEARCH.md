## Summary

Phase 27.9 adds a dedicated Inbox tab — a triage queue showing unassigned action items, recently-captured unprocessed notes, and empty notes. The user can dismiss, assign, delete, or backlink items without leaving the inbox, using a split-view layout (list left, detail pane right).

This phase is a standard addition to an already mature page pattern. Six prior page phases (People, Meetings, Projects, Intelligence, Actions, Notes) have established a stable wiring formula: add view to UIContext `View` union, add tab to `TABS` array in `TabBar.tsx`, add branch in `App.tsx`, create `InboxPage.tsx`, add Flask endpoint, add DB migration. The research confirms all components needed are already in the codebase.

The only net-new infrastructure is: (1) the `/inbox` API endpoint with three SQL queries, (2) a dismiss persistence mechanism (one DB table), (3) a backlink-creation flow reusing the existing `relationships` INSERT pattern, and (4) the InboxPage React component with split-view layout.

**Primary recommendation:** Implement dismiss as a `dismissed_inbox_items (path TEXT, dismissed_at TEXT)` table — simpler than a flag on notes, no schema coupling, trivially migratable. Backlink picker: debounce 300ms, min 2 chars, max 10 results, reuse GET /search endpoint.

### SQL Queries

**Unassigned actions (with dismiss filter):**
```sql
SELECT a.id, a.note_path, a.text, a.created_at
FROM action_items a
LEFT JOIN dismissed_inbox_items d ON d.path = CAST(a.id AS TEXT) AND d.item_type = 'action'
WHERE a.done = 0
  AND a.assignee_path IS NULL
  AND d.path IS NULL
  -- optional: AND a.note_path = :source_note
ORDER BY a.created_at DESC
LIMIT 20 OFFSET :offset
```

**Unprocessed notes:**
```sql
SELECT n.path, n.title, n.created_at
FROM notes n
LEFT JOIN relationships r ON r.source_path = n.path OR r.target_path = n.path
LEFT JOIN dismissed_inbox_items d ON d.path = n.path AND d.item_type = 'note'
WHERE n.type IN ('ideas', 'idea', 'coding', 'strategy', 'personal', 'projects')
  AND (n.tags IS NULL OR n.tags = '[]' OR n.tags = 'null')
  AND n.created_at >= datetime('now', '-14 days')
  AND r.source_path IS NULL
  AND d.path IS NULL
ORDER BY n.created_at DESC
LIMIT 50
```

Note: `type = 'ideas'` vs `'idea'` — check actual stored types. The `create_note` endpoint in api.py uses `subdir = "ideas" if note_type == "idea" else note_type` and stores the type as passed. Captured notes may have `type: idea` (singular). Query should include both `'idea'` and `'ideas'` to be safe, or check actual data.

**Empty notes (existing function, add dismiss filter):**
```sql
SELECT path, title FROM notes
WHERE (body IS NULL OR TRIM(body) = '')
  AND path NOT IN (SELECT path FROM dismissed_inbox_items WHERE item_type = 'note')
LIMIT 20
```

### Dismiss Persistence

**Recommended: `dismissed_inbox_items` table** (not a flag on notes).

Rationale: a flag on notes couples the inbox concern to the notes table and requires a migration that touches the core schema; a separate table is isolated, easy to purge, and handles heterogeneous item types (notes and actions).

```sql
CREATE TABLE IF NOT EXISTS dismissed_inbox_items (
    path      TEXT NOT NULL,        -- note path OR CAST(action_id AS TEXT)
    item_type TEXT NOT NULL,        -- 'note' | 'action'
    dismissed_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
    PRIMARY KEY (path, item_type)
);
```

Migration function added to `engine/db.py` as `migrate_add_dismissed_inbox_items_table(conn)`, called from `init_schema()`.

**Dismiss endpoint:**
```python
@app.post("/inbox/dismiss")
def dismiss_inbox_item():
    data = request.get_json(force=True) or {}
    path = data["path"]        # note path or str(action_id)
    item_type = data["item_type"]  # 'note' | 'action'
    conn = get_connection()
    conn.execute(
        "INSERT OR IGNORE INTO dismissed_inbox_items (path, item_type) VALUES (?, ?)",
        (path, item_type)
    )
    conn.commit()
    conn.close()
    return jsonify({"dismissed": True})
```

### Backlink Creation (reuse existing pattern)
The `relationships` table and INSERT pattern are established. Add a thin endpoint:

```python
@app.post("/relationships")
def create_relationship():
    data = request.get_json(force=True) or {}
    source = data["source_path"]
    target = data["target_path"]
    conn = get_connection()
    conn.execute(
        "INSERT OR IGNORE INTO relationships (source_path, target_path, rel_type) VALUES (?, ?, 'link')",
        (source, target)
    )
    conn.commit()
    conn.close()
    return jsonify({"created": True})
```

This also signals to remove the note from Unprocessed on the next inbox fetch (because the relationship now exists).

### Anti-Patterns to Avoid
- **Separate dismiss flags on `notes` table:** Adds inbox-specific state to core schema; harder to clear later.
- **Fetching all notes then filtering in React:** The unprocessed query is a DB join; do it in SQL.
- **Using `currentView === 'notes'` guards for sidebar/RightPanel around inbox:** The Inbox split-view has its own detail pane; `RightPanel` must NOT render for inbox view — add `currentView === 'inbox'` as a no-render condition in App.tsx.
- **Reusing NoteContext openNote() for the detail pane:** The split-view detail pane selection is local inbox state, not global NoteContext state. Use local `useState<InboxItem | null>` to avoid side-effects on the Notes tab state.

## Common Pitfalls

### Pitfall 1: Section Always-Visible at Count 0
**What goes wrong:** Developer wraps sections in `{count > 0 && <Section>}` — sections disappear when empty, breaking the locked UX decision.
**How to avoid:** Always render all three `<Section>` components. The count badge shows 0. Inner content shows an empty-state message ("Nothing to triage here.").
**Warning signs:** Any conditional `&&` around a Section render.

### Pitfall 2: note type filter — 'idea' vs 'ideas'
**What goes wrong:** `api.py` create_note uses `subdir = "ideas" if note_type == "idea" else note_type` but stores `note_type` (the value passed in, e.g. `"idea"`). Brain notes captured via CLI may store `type: idea` in frontmatter. The unprocessed query must include both `'idea'` and `'ideas'` or confirm what's actually in the DB before shipping.
**How to avoid:** Query includes both: `type IN ('idea', 'ideas', 'coding', 'strategy', 'personal', 'projects')`.

### Pitfall 3: RightPanel Renders Alongside InboxPage
**What goes wrong:** App.tsx renders `{currentView === 'notes' && <RightPanel />}` — this correctly hides it for non-notes views. But if the inbox view needs its own split-view detail pane, and RightPanel were added for inbox too, there would be two panes. Confirm RightPanel condition stays `notes`-only.
**How to avoid:** Do NOT add `'inbox'` to the RightPanel condition. The inbox detail pane is self-contained inside InboxPage.

### Pitfall 4: Stale Inbox After Inline Action
**What goes wrong:** User dismisses or assigns an item; the inbox list doesn't update, so the item still appears.
**How to avoid:** After any inline action (dismiss, assign, delete, add backlink), call `loadInbox()` to refetch. Use `useCallback` for loadInbox and include it in action handlers.

### Pitfall 5: Tab/App.tsx Must Import InboxPage Before Build
**What goes wrong:** Committing TabBar.tsx change without InboxPage.tsx causes TypeScript build failure.
**How to avoid:** Tasks 1+2 of the frontend plan committed together — same pattern as Phase 27.4, 27.5, 27.6.
**Rule from STATE.md:** "Tasks 1+2 committed together — App.tsx imports InboxPage so both must exist before build verifies."

### Pitfall 6: Test Isolation — DB_PATH Not Patched
**What goes wrong:** Tests call `get_connection()` without patching `engine.db.DB_PATH` and `engine.paths.DB_PATH`, causing writes to real `~/SecondBrain/.meta/brain.db`.
**How to avoid:** Any test fixture for inbox endpoint must patch both `engine.db.DB_PATH` and `engine.paths.DB_PATH`. Follow the pattern in `test_people.py` / `test_gui.py` `client` fixture.

### Pitfall 7: Deploy Pipeline After Frontend Changes
**What goes wrong:** Frontend change appears in source but is invisible in running GUI (stale installed binary).
**How to avoid:** After every `frontend/src/**` change: `npm run build` → `uv tool install . --reinstall` → restart `sb-api`. This is mandatory per LEARNINGS.md.

### DB Migration (engine/db.py pattern)
```python
def migrate_add_dismissed_inbox_items_table(conn: sqlite3.Connection) -> None:
    """Idempotent migration: create dismissed_inbox_items table if absent."""
    conn.execute("""
        CREATE TABLE IF NOT EXISTS dismissed_inbox_items (
            path         TEXT NOT NULL,
            item_type    TEXT NOT NULL,
            dismissed_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
            PRIMARY KEY (path, item_type)
        )
    """)
    conn.commit()
```
