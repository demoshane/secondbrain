## Summary

Phase 27.5 is a direct structural clone of Phase 27.4 (People Page), substituting `meetings/` notes for `people/` notes. The pattern is fully established and verified in working code: a split-panel page (left = directory list, right = detail view), a new `/meetings` Flask endpoint, a new `MeetingSummary` type, and wiring into `TabBar` and `App.tsx`. No new libraries or migrations are needed.

Meeting notes already exist as a first-class note type stored in `~/SecondBrain/meetings/` with YAML frontmatter. The SQLite `notes` table already has `type`, `path`, `title`, `updated_at` columns — the same columns used by the people query. Participants are stored in the `people` frontmatter column (JSON array), already indexed in the DB. Action items are linked via `note_path` on the `action_items` table.

**Primary recommendation:** Copy the People Page structure verbatim, replace `person`→`meeting`, add a `participants` column display, and sort by date descending instead of title ascending.

### Pattern 2: Flask List Endpoint (from /people)
**What:** Single SQL query joining `notes` with a correlated subquery for action count; `ORDER BY` column differs (date vs title).
**Example:**
```python
# Source: engine/api.py:249-258 (people endpoint)
@app.get("/meetings")
def list_meetings():
    conn = get_connection()
    conn.row_factory = sqlite3.Row
    rows = conn.execute(
        "SELECT n.path, n.title, substr(n.created_at,1,10) AS meeting_date, "
        "  n.people, "
        "  (SELECT COUNT(*) FROM action_items a WHERE a.note_path=n.path AND a.done=0) AS open_actions "
        "FROM notes n WHERE n.type = 'meeting' ORDER BY n.created_at DESC",
    ).fetchall()
    conn.close()
    result = []
    for r in rows:
        people_raw = r["people"] or "[]"
        try:
            participants = json.loads(people_raw)
        except Exception:
            participants = []
        result.append({
            "path": r["path"],
            "title": r["title"],
            "meeting_date": r["meeting_date"],
            "participant_count": len(participants),
            "open_actions": r["open_actions"],
        })
    return jsonify({"meetings": result})
```

### Pattern 4: Action Items Query for a Meeting
**What:** `GET /actions?note_path=<enc>` — action items are linked by `note_path`, not `assignee_path` (that is the people pattern).
**Why it differs from People:** People page queries `assignee_path`; Meetings page queries `note_path` because actions belong to the meeting note.

### Anti-Patterns to Avoid
- **Querying actions by assignee_path for meetings:** Meetings link actions via `note_path`, not `assignee_path`. Using the wrong column returns no results.
- **Sorting meetings by title:** Sort by `created_at DESC` so newest meetings appear first — this is a chronological log, not a directory.
- **Using `type = 'meetings'` (plural):** The note type stored in frontmatter and DB is `'meeting'` (singular). Verify against actual note files before writing the query.
- **Assuming participants are in a `participants` column:** Participants are stored in the `people` frontmatter column (JSON array), which maps to the `people` column in the `notes` table. No new column needed.

## Common Pitfalls

### Pitfall 1: Wrong action items query column
**What goes wrong:** Page shows 0 action items even when meeting notes have actions.
**Why it happens:** Copying People page code which uses `assignee_path`; meetings use `note_path`.
**How to avoid:** Query `GET /actions?note_path=<enc>` not `?assignee=<enc>`.
**Warning signs:** Actions section always shows "No open actions" even for meetings with known actions.

### Pitfall 2: `people` column is JSON string, not array
**What goes wrong:** Participant count shows `NaN` or component crashes.
**Why it happens:** `notes.people` is stored as a JSON string (e.g. `'["Alice", "Bob"]'`), not a native array.
**How to avoid:** Parse with `json.loads()` in Python (with `[]` fallback); in TypeScript use `JSON.parse()` or expect the API to return a pre-parsed array.
**Decision:** Have the backend return `participant_count: int` and `participants: list[str]` — avoids JSON parsing in the frontend.

### Pitfall 3: Meeting note type mismatch
**What goes wrong:** `/meetings` endpoint returns empty list despite meeting notes existing on disk.
**Why it happens:** The `type` field in frontmatter may be `'meetings'`, `'meeting'`, or absent depending on how notes were captured. The DB `type` column reflects frontmatter.
**How to avoid:** Run a quick DB check before writing the query: `SELECT DISTINCT type FROM notes WHERE path LIKE '%/meetings/%'`. Document the actual value in the plan.
**Warning signs:** Empty list with no error; confirmed notes in `~/SecondBrain/meetings/` folder.

### Pitfall 4: App.tsx view routing is a cascade of ternaries
**What goes wrong:** Adding `MeetingsPage` to the wrong branch, or forgetting to update the `else` fallback.
**Why it happens:** The current App.tsx uses nested ternaries; the `people` branch is the final `else`. Adding `meetings` must replace `else` with `else if (people) ... else (meetings)`.
**How to avoid:** Follow the exact ternary chain pattern; the App.tsx section is small and clearly structured.

### Pitfall 5: Playwright test fixture must seed a meeting note
**What goes wrong:** Playwright test_meetings.py row-click tests fail because the meetings list is empty.
**Why it happens:** Same issue as 27.4-04 people fixture. The test DB is isolated per session.
**How to avoid:** Seed a `test-meeting.md` file and SQLite row at session scope in `conftest.py`, mirroring the `gui_brain` fixture from test_gui.py.

## Open Questions

1. **Actual `type` value for meeting notes in DB**
   - What we know: Folder is `meetings/`; capture templates use `type: meeting`
   - What's unclear: Whether all existing notes have `type: meeting` set in frontmatter or if some are missing the field
   - Recommendation: Plan 27.5-01 (Wave 0 stubs) should include a DB introspection check; query `SELECT DISTINCT type FROM notes WHERE path LIKE '%/meetings/%'` and document result before writing the endpoint

2. **Whether `GET /actions` supports `note_path` filter param**
   - What we know: `GET /actions` supports `assignee` filter (confirmed from 27.4 decisions); `list_actions()` in `intelligence.py` is the underlying function
   - What's unclear: Whether `note_path` filter is already implemented or needs to be added
   - Recommendation: Check `list_actions()` signature before writing backend plan; add `note_path` param if missing (small addition to existing conditional SQL pattern)
