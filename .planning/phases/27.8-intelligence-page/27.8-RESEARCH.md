# Phase 27.8: Intelligence Page - Research

**Researched:** 2026-03-18
**Domain:** React/TypeScript frontend page promotion; Flask API; brain intelligence features
**Confidence:** HIGH

## Summary

Phase 27.8 promotes the intelligence panel that already exists in `RightPanel.tsx` into a dedicated first-class tab page. This is a promotion/extraction task, not a greenfield feature. All backend endpoints already exist and are production-ready. The frontend already has all the data-fetching logic in RightPanel; IntelligencePage is a reorganisation of that logic into a full-page layout.

The pattern is identical to People (27.4), Meetings (27.5), and Projects (27.6): add a tab ID to UIContext's View union, add an entry to the TABS array in TabBar, create a new page component, wire it into App.tsx, and add two Playwright smoke tests.

The intelligence content has three distinct subsections: Recap (on-demand, AI-generated, POST trigger), Brain Health (rich detail dashboard with score + breakdown tables), and Digest/Stale nudges (from GET /intelligence). The full-page layout should give each subsection the space it deserves — unlike the cramped 256 px right-panel.

**Primary recommendation:** Follow the Projects page layout pattern exactly. No new backend endpoints are required. The only backend consideration is whether to expose the full brain-health breakdown (orphans list, broken links list, duplicates list) — the endpoint already returns all of this; IntelligencePage just needs to render it.

### Pattern 4: API Shape — What each endpoint returns
**GET /intelligence** returns:
```json
{ "recap": null, "nudges": [{ "path": "...", "title": "...", "updated_at": "..." }] }
```
Note: `recap` is always null from GET (it is never cached server-side). `connections` and `digest` keys referenced in RightPanel but not returned by current implementation — RightPanel fetches them from `/intelligence` response but they are absent (defaults to empty string/array silently).

**POST /intelligence/recap** returns:
```json
{ "recap": "...generated text..." }
```
Can return 500 with `{ "recap": "Error: ..." }`.

**GET /brain-health** returns:
```json
{
  "score": 87,
  "total_notes": 142,
  "orphan_count": 3,
  "orphans": [{ "path": "...", "title": "..." }, ...],  // max 20
  "empty_count": 1,
  "empty_notes": [{ "path": "...", "title": "..." }, ...],
  "broken_link_count": 2,
  "broken_links": [{ "source": "...", "target": "..." }, ...],
  "duplicate_count": 0,
  "duplicate_candidates": [{ "a": "...", "b": "...", "similarity": 0.94 }, ...]
}
```
RightPanel only showed `score`. IntelligencePage can render all breakdown lists.

### Anti-Patterns to Avoid
- **Removing intelligence from RightPanel:** The right panel still shows Backlinks and People sections which are note-context-specific. Only the global intelligence sections (Recap, Brain Health, Digest) are promoted. Do NOT delete RightPanel or the recap/health sections from it entirely — assess whether to keep a compact summary in RightPanel or strip it. Consensus pattern: remove the promoted sections from RightPanel to avoid duplication.
- **Adding a /intelligence/digest endpoint:** Digest is currently unimplemented (GET /intelligence returns `recap: null` and no `digest` key). Do not build digest in this phase unless it is in scope.
- **Blocking the UI on recap generation:** generateRecap() already handles loading state (generatingRecap flag + animate-spin) — transplant the pattern from RightPanel, don't redesign it.

## Common Pitfalls

### Pitfall 1: App.tsx implicit else branch
**What goes wrong:** `<ProjectsPage />` is currently rendered by a final `else` — adding Intelligence requires making Projects explicit first or Intelligence will never render.
**Why it happens:** Incremental page additions without refactoring the else chain.
**How to avoid:** Make the Projects branch explicit when adding the Intelligence branch (one-line change).
**Warning signs:** Intelligence tab click shows ProjectsPage content.

### Pitfall 2: Frontend deploy pipeline not run
**What goes wrong:** Tab does not appear in running GUI despite correct source changes.
**Why it happens:** Stale installed binary (see LEARNINGS.md — this has happened twice already on 27.4 and 27.6).
**How to avoid:** Always run full deploy pipeline: `npm run build` → `uv tool install . --reinstall` → restart sb-api.
**Warning signs:** Old JS bundle hash in `~/.local/share/uv/tools/second-brain/.../index.html`.

### Pitfall 3: data-testid missing from page root
**What goes wrong:** Playwright `wait_for_selector('[data-testid="intelligence-page"]')` fails.
**Why it happens:** Forgetting to add `data-testid` to the root div.
**How to avoid:** Root div of every page component has `data-testid="<page-name>-page"` — confirmed from all prior pages.

### Pitfall 4: generate_recap errors on empty brain
**What goes wrong:** POST /intelligence/recap returns 500 in test environment with empty DB.
**Why it happens:** `generate_recap_on_demand()` calls AI adapter; empty DB returns "No notes captured in last 7 days." — this is a safe fallback, not an error. The 500 only fires on unexpected exception.
**How to avoid:** Test the generate-recap button visibility, not the generated content.

### Pitfall 5: Brain health score mismatches compute_health_score signature
**What goes wrong:** Calling `compute_health_score(total, orphans, broken, duplicates)` with wrong parameter names.
**Why it happens:** The function signature uses keyword args `orphans`, `broken`, `duplicates` — the api.py call uses `empty` instead of `orphans` for `empty_count`. Check current call site.
**How to avoid:** Read the actual call in `api.py` lines 879-884 before writing any test that hits the endpoint.

### Pitfall 6: get_empty_notes not in brain_health module
**What goes wrong:** `get_empty_notes` is imported in api.py but not defined in brain_health.py (only `get_orphan_notes`, `get_duplicate_candidates`, `compute_health_score` are defined there).
**Why it happens:** `get_empty_notes` may be defined elsewhere or may have been removed. The api.py import line 870 references it — but the brain_health.py file only has 3 functions.
**How to avoid:** Verify this import works before assuming the `/brain-health` endpoint is fully functional. If `get_empty_notes` is missing, the endpoint returns 500.
**Warning signs:** `/brain-health` returns `{"error": "..."}` 500 in tests.

## Open Questions

1. **Should Recap/Health sections remain in RightPanel after promotion?**
   - What we know: RightPanel will become notes-only (backlinks + people) after extraction
   - What's unclear: User preference for quick-glance score in right panel vs. clean separation
   - Recommendation: Remove from RightPanel to avoid duplication; the Intelligence tab is one click away

2. **Is `get_empty_notes` actually importable from `engine.brain_health`?**
   - What we know: `api.py` line 870 imports it; `brain_health.py` only defines 3 functions (get_orphan_notes, get_duplicate_candidates, compute_health_score)
   - What's unclear: Whether it exists in another module or is a dead import
   - Recommendation: Verify `GET /brain-health` actually returns 200 before writing tests. If broken, fix the import as part of Wave 0.

3. **Digest feature scope**
   - What we know: RightPanel has digest state (`const [digest, setDigest] = useState('')`) but GET /intelligence never returns a digest key. The section only renders when `digest` is truthy (never).
   - What's unclear: Whether digest should be added to IntelligencePage or remain dormant
   - Recommendation: Do not implement digest generation in this phase. Render a placeholder "Digest — coming soon" or omit entirely.
