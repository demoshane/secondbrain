# Phase 27.1: Smart Capture & Proactive Brain - Research

**Researched:** 2026-03-17
**Domain:** Capture pipeline enrichment, embedding-based dedup, MCP tool self-documentation, project documentation
**Confidence:** HIGH

## Summary

Phase 27.1 adds five independent features on top of the fully-working capture + MCP stack. The codebase already has all the required infrastructure: `capture_note()` in `engine/capture.py` is the single write path, `embed_texts()` in `engine/embeddings.py` produces 384-float BLOBs for sqlite-vec, FastMCP 3.x exposes `t.parameters` and `t.description` on every tool object via `mcp.list_tools()`, and `build_post()` already produces a structured frontmatter dict where new entity fields can be added.

Entity extraction can be done purely with regex (people already exist in frontmatter; places and topics need extraction from body text). A lightweight regex + heuristic approach is preferred over an AI call at capture time to avoid latency and dependency on network availability. Dedup uses the existing `note_embeddings` table and `vec_distance_cosine` — the same pattern used by `find_similar()` in `intelligence.py`. Multi-capture is a thin wrapper in `mcp_server.py` that calls `capture_note()` in a loop within a single transaction. CLAUDE.md is a static write. MCP self-documentation is a new `sb_tools` tool that calls `asyncio.run(mcp.list_tools())` and returns structured dicts.

**Primary recommendation:** Implement all five features as separate plans; entity extraction and dedup are the most complex and should each get a dedicated plan with test scaffold.

### Pattern 1: Entity Extraction at Capture Time
**What:** `extract_entities(title, body)` returns `{"people": [...], "places": [...], "topics": [...]}`. Called inside `capture_note()` after `build_post()`, before `write_note_atomic()`. Results stored in frontmatter `entities` field and in a new `entities` TEXT column (JSON) in the `notes` table.

**When to use:** Every call to `capture_note()` — CLI and MCP.

**Extraction heuristics (regex, no AI):**
- **People**: Capitalized "First Last" pairs (2+ words, both Title Case), excluding known stop words ("The", "This", "When", etc.); merge with existing `people` list.
- **Topics**: Words starting with `#` in body text (hashtag-style); capitalized nouns repeated 2+ times.
- **Places**: Capitalized tokens preceded by "in", "at", "from", "to" that are not people names.

```python

def extract_entities(title: str, body: str) -> dict:
    """Best-effort entity extraction. Never raises. Returns {"people": [], "places": [], "topics": []}."""
    try:
        ...
    except Exception:
        return {"people": [], "places": [], "topics": []}
```

### Pattern 2: Embedding-Based Dedup Check
**What:** Before writing, embed the incoming note title+body, query `note_embeddings` via `vec_distance_cosine`, return candidates above threshold.

**When to use:** In MCP `sb_capture()` tool only (not CLI capture — CLI is interactive and can prompt; MCP is the primary agent interface). If similar notes found above threshold (0.92 suggested — higher than `find_similar`'s 0.8 to avoid false positives at capture time), return `{"status": "duplicate_warning", "similar": [...], "confirm_token": tok}`. Second call with `confirm_token` bypasses check and saves.

**Key constraint:** Embedding requires the model to be loaded. Use `best-effort` pattern — if embedding fails (model not loaded, no embeddings in table), proceed with capture normally.

```python
# Source: find_similar() pattern in engine/intelligence.py
def check_capture_dedup(title: str, body: str, conn, threshold: float = 0.92) -> list[dict]:
    """Return list of similar notes if dedup threshold exceeded. Empty list = safe to capture."""
    try:
        from engine.embeddings import embed_texts
        blobs = embed_texts([f"{title}\n{body}"])
        if not blobs:
            return []
        # query note_embeddings via vec_distance_cosine
        ...
    except Exception:
        return []  # best-effort — never block capture
```

### Pattern 3: Batch Capture (MCP)
**What:** `sb_capture_batch(notes: list[dict])` takes a list of `{title, body, note_type, tags, sensitivity}` dicts. Runs each through `capture_note()` in a single `get_connection()` context (one conn, N writes, one final commit). Returns list of `{status, path, index}` or `{status: "error", reason, index}` per note.

**When to use:** Agent workflows where multiple captures are logically atomic (e.g., end-of-meeting: capture meeting note + capture 3 action notes in one call).

**Transaction semantics:** Best-effort per-note — if one note fails, others still succeed (skip, don't abort). Return partial results. This is consistent with the existing `batch_capture` in Phase 25 which returns `{succeeded: [...], failed: [...]}`.

### Pattern 4: MCP Tool Self-Documentation (sb_tools)
**What:** New `sb_tools()` MCP tool. Calls `asyncio.run(mcp.list_tools())` and returns a list of structured dicts. No DB access needed.

**FastMCP 3.x tool object fields (verified):**
- `t.name` — tool name string
- `t.description` — docstring
- `t.parameters` — JSON Schema dict `{type: "object", properties: {...}, required: [...]}`
- `t.output_schema` — JSON Schema dict for return type

```python
# Source: verified via uv run python inspection of FastMCP 3.x
@mcp.tool()
def sb_tools() -> list[dict]:
    """List all available MCP tools with their schemas. Use this to discover what operations are available."""
    import asyncio
    tools = asyncio.run(mcp.list_tools())
    return [
        {
            "name": t.name,
            "description": t.description or "",
            "parameters": t.parameters,
            "output_schema": t.output_schema,
        }
        for t in tools
        if t.name != "sb_tools"  # exclude self to avoid infinite recursion in agent loops
    ]
```

### Anti-Patterns to Avoid
- **AI call at capture time for entity extraction:** Adds latency and online dependency. Use regex instead.
- **Blocking capture on dedup failure:** If embedding or DB lookup fails, always fall through to capture. Best-effort only.
- **Aborting batch on first failure:** Return partial results; each note is independent.
- **Running `asyncio.run()` inside an existing event loop:** `sb_tools` is called from stdio MCP transport which runs in an asyncio loop. Use `asyncio.get_event_loop().run_until_complete()` or check `mcp._tool_manager` directly instead. Verify at implementation time.

## Common Pitfalls

### Pitfall 1: asyncio.run() inside running event loop
**What goes wrong:** `sb_tools` calls `asyncio.run(mcp.list_tools())` but FastMCP's stdio transport already runs an event loop. `asyncio.run()` raises `RuntimeError: This event loop is already running`.
**Why it happens:** FastMCP runs the server in an async context.
**How to avoid:** Access `mcp._tool_manager._tools` directly (a dict) or use `mcp.get_tool(name)` — investigate the non-async path at implementation time. Alternative: use `asyncio.get_event_loop().run_until_complete()` only if loop is not running, else use `nest_asyncio`.
**Warning signs:** `RuntimeError: This event loop is already running` in tests.

### Pitfall 2: Entity extraction false positives
**What goes wrong:** Regex captures "The Note", "This Meeting" as people names.
**Why it happens:** Generic Title Case text matches the "First Last" heuristic.
**How to avoid:** Maintain a stop-word list of common Title Case non-names. Keep extraction conservative (two consecutive Title Case words, not just one).

### Pitfall 3: Dedup threshold too low causes captures to be blocked
**What goes wrong:** Threshold of 0.8 (same as find_similar) treats many legitimate new notes as duplicates.
**Why it happens:** Short notes have high cosine similarity on topically-related content.
**How to avoid:** Use 0.92 threshold. Test against real brain data. Never block on dedup; always offer confirm path.

### Pitfall 4: entities column missing on existing DBs
**What goes wrong:** `notes` table has no `entities` column on upgraded DBs.
**Why it happens:** Schema was created before this phase.
**How to avoid:** Follow existing pattern — add `migrate_add_entities_column()` in `engine/db.py`, call it from `init_schema()`.

### Pitfall 5: Multi-capture partial failures leave DB dirty
**What goes wrong:** Note 3 of 5 fails; notes 1-2 were committed, 4-5 not attempted.
**Why it happens:** Each `capture_note()` calls `conn.commit()` internally.
**How to avoid:** Accept per-note isolation (consistent with Phase 25 batch). Document in `sb_capture_batch` docstring that partial success is expected. Return `{succeeded: [...], failed: [...]}` like Phase 25.

## Open Questions

1. **asyncio.run() inside FastMCP transport event loop**
   - What we know: `asyncio.run()` raises if an event loop is already running; FastMCP stdio runs async.
   - What's unclear: Whether `mcp._tool_manager._tools` is a public-enough attribute to use directly.
   - Recommendation: At implementation time, check `mcp._tool_manager._tools.values()` first; fall back to `nest_asyncio` if needed.

2. **Dedup at CLI capture time**
   - What we know: CLI capture is interactive; dedup would require printing a warning and prompting.
   - What's unclear: Whether the user wants dedup in CLI too (success criteria only mentions "capturing a note" generically).
   - Recommendation: Implement dedup in `sb_capture` (MCP) only for Phase 27.1. CLI can be added in a follow-up.

3. **entities column vs frontmatter-only storage**
   - What we know: Other structured data (tags, people) is stored both in frontmatter and in a DB column.
   - What's unclear: Whether entities need to be queryable via SQL for future phases (People Page, search).
   - Recommendation: Store in both frontmatter (for portability) and DB column (for future queries). Follows existing pattern.
