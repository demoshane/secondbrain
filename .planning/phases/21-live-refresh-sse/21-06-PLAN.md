```javascript
let currentPath = null;
```

enterEditMode() — sets easyMDE, sets isDirty=false, attaches change listener that sets isDirty=true
exitEditMode() — destroys easyMDE (easyMDE=null), hides editor UI
openNote(path) — sets currentPath, calls exitEditMode() internally, renders viewer

Current handleNoteEvent (lines 266-290):
```javascript
function handleNoteEvent({ type, path }) {
    loadNotes();
    if (!currentPath) return;
    const matchesCurrent = currentPath.endsWith('/' + path) || currentPath === path;
    if (!matchesCurrent) return;
    if (type === 'deleted') {
        currentPath = null;
        document.getElementById('viewer').innerHTML = '<em>Note was deleted.</em>';
        exitEditMode();
        return;
    }
    if (type === 'modified' || type === 'created') {
        if (isDirty) {
            showConflictBanner(currentPath);
        } else {
            openNote(currentPath);
        }
    }
}
```

Root cause: isDirty alone is insufficient — easyMDE could be null (editor closed) while isDirty is
stale-true from a previous session. Conversely, if a fast external event arrives immediately after
enterEditMode() but before the first keystroke, isDirty is false and easyMDE is non-null but the
user IS in the editor. The correct semantic is: show banner if the editor is currently open
(easyMDE !== null) regardless of isDirty, because any open editor session must not be
silently destroyed.


Replace with:
```javascript
if (easyMDE !== null) {
    // Editor is open — always show conflict banner regardless of isDirty.
    // Any open editor session must not be silently destroyed by an external event.
    // The user can choose Keep (dismiss banner) or Load (reload and close editor).
    showConflictBanner(currentPath);
} else if (isDirty) {
    // isDirty but editor already closed (defensive: treat as conflict)
    showConflictBanner(currentPath);
} else {
    openNote(currentPath);
}
```

This ensures:
- If the editor is open (easyMDE !== null), the banner always appears — even before the first keystroke (isDirty still false). The user consciously clicked Edit, so destroying their session without confirmation is wrong regardless of dirty state.
- If the editor is closed but isDirty is stale-true for some reason, banner still shows as defensive fallback.
- If neither, auto-reload as before (viewer-only mode).
