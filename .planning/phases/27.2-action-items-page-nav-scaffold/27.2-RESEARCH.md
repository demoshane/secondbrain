## Summary

Phase 27.2 adds a tab navigation scaffold to the existing single-panel Flask/pywebview GUI, then builds a full Action Items page on that scaffold. The codebase is vanilla JS ES modules with no framework ‚Äî all new UI follows the existing fetch-then-innerHTML render cycle.

The implementation has three distinct work tracks that can be staged: (1) DB migration adding `assignee_path` and `due_date` columns, (2) API extension adding filter params to `GET /actions` and a new `PUT /actions/<id>` for assignee writes, and (3) frontend ‚Äî tab bar, Actions page table, filter controls, assignee picker, and the My Actions sidebar section.

The existing `loadActions()` function and `POST /actions/<id>/done` endpoint provide the core patterns to extend. The migration pattern from `migrate_add_attachments_table()` is the exact model for the two new columns.

**Primary recommendation:** Stage as three waves ‚Äî migration first, then API, then frontend ‚Äî so each wave is independently testable.

### Pattern 1: DB Migration (migrate_add_* idiom)
**What:** Idempotent `ALTER TABLE` guarded by `PRAGMA table_info` column check.
**When to use:** Any new column on an existing table.
**Example (from db.py):**
```python
# Source: engine/db.py migrate_add_people_column
def migrate_add_assignee_path(conn):
    cols = {row[1] for row in conn.execute("PRAGMA table_info(action_items)").fetchall()}
    if "assignee_path" not in cols:
        conn.execute("ALTER TABLE action_items ADD COLUMN assignee_path TEXT NULL")
        conn.commit()
```
Both `migrate_add_assignee_path` and `migrate_add_due_date` must be called from `init_schema()`.

### Pattern 2: Tab Bar State Machine
**What:** Module-level `currentView` variable; `showView(id)` toggles CSS `display` on view containers.
**When to use:** Any view switch; future tabs follow the same pattern.
**Example:**
```javascript
// Source: established project pattern (vanilla JS)
const TABS = [{id: 'notes', label: 'Notes'}, {id: 'actions', label: 'Action Items'}];
let currentView = 'notes';

function showView(id) {
    currentView = id;
    document.querySelectorAll('.tab-btn').forEach(btn => {
        btn.classList.toggle('active', btn.dataset.tab === id);
    });
    document.getElementById('layout').style.display = id === 'notes' ? '' : 'none';
    document.getElementById('actions-page').style.display = id === 'actions' ? '' : 'none';
}
```
Notes tab: show `#layout`, hide `#right-panel` (per decision). Actions tab: show `#actions-page`, hide `#layout`.

### Pattern 3: Table Render (fetch-then-innerHTML)
**What:** Fetch data, build HTML string via `.map()`, assign to `element.innerHTML`.
**When to use:** All list/table rendering ‚Äî existing pattern throughout app.js.
**Example (extending loadActions pattern):**
```javascript
// Source: engine/gui/static/app.js loadActions()
async function renderActionsPage() {
    const params = new URLSearchParams({ done: showDone ? '1' : '0' });
    if (activeAssigneeFilter) params.set('assignee', activeAssigneeFilter);
    const res = await fetch(`${API}/actions?${params}`);
    const { actions } = await res.json();
    const tbody = document.getElementById('actions-table-body');
    tbody.innerHTML = actions.map(a => `
        <tr data-id="${a.id}">
            <td><input type="checkbox" ${a.done ? 'checked' : ''} data-id="${a.id}"></td>
            <td>${a.text}</td>
            <td>${a.note_path ? a.note_path.split('/').pop().replace('.md','') : ''}</td>
            <td class="assignee-cell" data-id="${a.id}">${a.assignee_name || '<em>‚Äî</em>'}</td>
            <td>${a.created_at ? a.created_at.slice(0,10) : ''}</td>
        </tr>
    `).join('') || '<tr><td colspan="5"><em>No items</em></td></tr>';
    // attach checkbox + assignee click handlers
}
```

### Pattern 4: Inline Assignee Picker
**What:** Click on assignee cell ‚Üí absolute-positioned `<select>` dropdown populated from `_allNotes` (people type). Change event ‚Üí `PUT /actions/<id>` ‚Üí refresh row.
**When to use:** Single-cell inline editing ‚Äî consistent with tag chip inline edit pattern (Phase 23).
**Key detail:** Dismiss on `blur` or outside click. Use `position:absolute` relative to cell. Populate from `_allNotes.filter(n => n.type === 'people')` ‚Äî no extra API call since `_allNotes` is already loaded.

### Pattern 5: My Actions Section in loadMeta()
**What:** After existing people section renders, check if current note's type is `'people'`. If so, fetch `GET /actions?assignee=<path>&done=0` and render a new `<section>`.
**When to use:** Conditioned on `note.type === 'people'` check against `_allNotes` cache.
**Example:**
```javascript
// Source: engine/gui/static/app.js loadMeta() ‚Äî hook point after peopleList render
const cachedNote = _allNotes.find(n => n.path === path);
if (cachedNote && cachedNote.type === 'people') {
    const ar = await fetch(`${API}/actions?assignee=${encodeURIComponent(path)}&done=0`);
    const { actions } = await ar.json();
    const myActionsSection = document.getElementById('my-actions-section');
    myActionsSection.style.display = '';
    document.getElementById('my-actions-list').innerHTML =
        actions.map(a => `<li>${a.text}</li>`).join('') || '<li><em>None</em></li>';
}
```

### Anti-Patterns to Avoid
- **Extra API call for people list in assignee picker**: `_allNotes` is already cached ‚Äî filter it, don't fetch `GET /notes?type=people` on every cell click.
- **Modifying SCHEMA_SQL constant for new columns**: always use `migrate_add_*` functions, not in-place schema edits, to preserve idempotency on existing DBs.
- **Hiding `#right-panel` via JS visibility on Notes tab**: use `display:none` not `visibility:hidden` to avoid layout space being reserved.
- **Calling `loadActions()` (old right-panel version) from the Actions page tab**: the old `loadActions()` populates `#actions-list` in the right panel; the new `renderActionsPage()` is a separate function targeting `#actions-table-body`.

## Common Pitfalls

### Pitfall 1: Migration Not Called on Existing DB
**What goes wrong:** `migrate_add_assignee_path` and `migrate_add_due_date` functions exist but existing brain DB at `~/SecondBrain/.meta/brain.db` doesn't get the columns until `init_schema()` is called.
**Why it happens:** `init_schema()` is called at app startup via `get_connection()` in `api.py` ‚Äî but only if the call chain reaches it. Verify the call is in the right place.
**How to avoid:** Add both migration calls to `init_schema()` and run `uv run sb-reindex` or restart the server after the migration ships.
**Warning signs:** `OperationalError: table action_items has no column named assignee_path` in Flask logs.

### Pitfall 2: Notes Tab Right Panel Visibility
**What goes wrong:** On switching to Notes tab then back, `#right-panel` may stay hidden.
**Why it happens:** If `showView('notes')` hides `#right-panel` unconditionally, reopening a note must re-show it.
**How to avoid:** Re-show `#right-panel` inside `openNote()` (called when user clicks a note in Notes tab), not in `showView()`.

### Pitfall 3: `GET /actions` Filter Params Breaking Existing Callers
**What goes wrong:** Old `loadActions()` calls `GET /actions` with no params ‚Äî adding required params would break it.
**Why it happens:** Changing endpoint signature without backward-compat defaults.
**How to avoid:** All new params are optional with defaults (`done` defaults to `0`, `assignee` defaults to `None`/all). Existing `loadActions()` continues to work unchanged.

### Pitfall 4: Assignee Picker Dropdown Z-Index
**What goes wrong:** Inline dropdown hidden behind other elements.
**Why it happens:** `position:absolute` dropdown inside a `<td>` ‚Äî parent stacking context may clip it.
**How to avoid:** Append picker to `document.body` and position it using `getBoundingClientRect()`, or set `overflow:visible` on the table wrapper. Pattern seen in tag chip inline edit (Phase 23 used `position:absolute` on the chip container).

### Pitfall 5: `list_actions()` Signature Change Breaks test_api.py
**What goes wrong:** Extending `list_actions(conn, done=False)` to accept `assignee=None` is backward-compatible, but tests calling it directly need to verify the new parameter.
**Why it happens:** Test fixtures may assert exact row counts that change if assignee filter is applied.
**How to avoid:** New `assignee` param defaults to `None` (no filter). Existing tests pass unchanged. Add new tests for the filtered cases.

## Code Examples

### Extend `list_actions()` with filters
```python
# Source: engine/intelligence.py list_actions() ‚Äî extend signature
def list_actions(conn, done: bool = False, assignee: str | None = None) -> list[dict]:
    sql = "SELECT id, text, note_path, created_at, assignee_path FROM action_items WHERE done=?"
    params: list = [1 if done else 0]
    if assignee is not None:
        sql += " AND assignee_path=?"
        params.append(assignee)
    sql += " ORDER BY created_at DESC"
    rows = conn.execute(sql, params).fetchall()
    return [dict(r) for r in rows]
```

### Extend `GET /actions` endpoint
```python
# Source: engine/api.py GET /actions ‚Äî extend with query params
@app.get("/actions")
def get_actions():
    done = request.args.get("done", "0") == "1"
    assignee = request.args.get("assignee") or None
    conn = get_connection()
    conn.row_factory = sqlite3.Row
    actions = list_actions(conn, done=done, assignee=assignee)
    conn.close()
    return jsonify({"actions": actions})
```

### New `PUT /actions/<id>` for assignee write
```python
# Source: engine/api.py ‚Äî new endpoint, mirrors POST /actions/<id>/done pattern
@app.put("/actions/<int:action_id>")
def update_action(action_id):
    data = request.get_json(force=True)
    conn = get_connection()
    if "assignee_path" in data:
        conn.execute("UPDATE action_items SET assignee_path=? WHERE id=?",
                     (data["assignee_path"], action_id))
    conn.commit()
    conn.close()
    return jsonify({"updated": True, "id": action_id})
```

## Open Questions

1. **Should the right-panel `#actions-panel` still show on the Notes tab?**
   - What we know: decision says Notes tab hides `#right-panel` entirely (viewer-focused, more space)
   - What's unclear: if the user is viewing a note on Notes tab and wants to see its actions, they'd have to switch to the Actions tab and filter by source note
   - Recommendation: follow the decision (hide right panel on Notes tab); this is locked. The Actions tab with source note column satisfies the use case.

2. **`assignee_name` in API response vs. path join**
   - What we know: `action_items` stores `assignee_path` (e.g. `people/alice.md`); the table needs to display a human name
   - What's unclear: whether to JOIN with `notes` table to get `title`, or resolve name client-side from `_allNotes` cache
   - Recommendation: resolve client-side using `_allNotes.find(n => n.path === assignee_path)?.title` ‚Äî no JOIN needed, cache already loaded. If cache miss, show raw path basename as fallback.

3. **Filter dropdowns: populate assignee list from `GET /notes?type=people` or `_allNotes`?**
   - What we know: CONTEXT.md says "People options source: `GET /notes?type=people`" for the inline picker
   - What's unclear: same source for the filter dropdown?
   - Recommendation: use `_allNotes.filter(n => n.type === 'people')` for both ‚Äî consistent, zero extra fetch.
