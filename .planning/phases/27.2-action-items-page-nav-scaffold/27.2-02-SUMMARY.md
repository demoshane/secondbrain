# Phase 27.2 Plan 02: Action Items Backend — DB Migrations, Filters, and PUT Endpoint Summary

Backend fully wired for action items page: two idempotent DB migrations add `assignee_path` and `due_date` columns to `action_items`, `list_actions()` gains an `assignee` filter param, and the API gets query-param filtering plus a `PUT /actions/<id>` endpoint.

## What Was Built

### Task 1 — DB migrations (engine/db.py)

Two new migration functions added following the project's `migrate_add_*` idempotent ALTER TABLE pattern:

- `migrate_add_assignee_path(conn)` — adds `assignee_path TEXT NULL` to `action_items` if absent
- `migrate_add_due_date(conn)` — adds `due_date TEXT NULL` to `action_items` if absent

Both called at end of `init_schema()` after existing migrations. Existing DBs gain the columns transparently on next startup.

### Task 2 — list_actions() + API (engine/intelligence.py, engine/api.py)

`list_actions()` signature extended to `list_actions(conn, done=False, assignee=None)`:
- SELECT now includes `assignee_path` column
- `assignee` param conditionally appends `AND assignee_path=?` — backward-compatible, existing callers unaffected

API changes:
- `GET /actions` now reads `?done=0/1` and `?assignee=<path>` query params; defaults to `done=False, assignee=None`
- `PUT /actions/<id>` added; writes `assignee_path` field to the DB row
